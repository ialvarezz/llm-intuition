The Context Window

A model has no memory — only a window. Everything it knows about your conversation must fit inside, every single turn, and the window is always closer to full than you think.

The window is the world

Everything a model can see about your conversation — the system prompt, every prior message, any documents you've pasted in, and the reply it's currently writing — has to fit inside one fixed budget of tokens. Anthropic's own framing is blunt about what that budget actually is: "the 'context window' refers to all the text a language model can reference when generating a response, including the response itself. This is different from the large corpus of data the language model was trained on, and instead represents a 'working memory' for the model." That last phrase is the important one. The window isn't a cache of things the model has learned — it's the entire set of information available to it for this one response. Nothing outside it exists as far as the model is concerned. It's also a hard cap, not a soft guideline: feed a model more input tokens than its window holds and the request fails outright — the API "returns a 400 invalid_request_error ('prompt is too long') on every model" once the input alone exceeds the limit. There's no graceful degradation at the input stage; the window is a wall, not a slope.

Memory is re-sending

Chat apps feel like they remember you. They don't — the model has no memory between calls at all. What actually happens is that the app quietly re-sends the entire conversation so far, every single turn, and the model reads it fresh each time as if for the first time. This is why token counts keep climbing turn after turn: "as the conversation advances through turns, each user message and assistant response accumulates within the context window, and previous turns are preserved completely." Turn 20 of a conversation costs more to process than turn 1, because turn 20's request silently includes turns 1 through 19 as well. The illusion of a model that recalls your name from three messages ago is really an app faithfully replaying the whole transcript back to a model that has no memory of ever seeing it. And at the limit, chat interfaces quietly work around a full window rather than erroring out — "chat interfaces such as claude.ai can also manage the context window on a rolling 'first in, first out' basis." The oldest turns fall off the back to make room for new ones, with nothing in the UI announcing it — you just notice the model no longer reacts to something from earlier.

The KV cache

When a model generates a token, attention doesn't just look at the newest word — it recomputes relationships against every earlier token in the context, every single time. Doing that from scratch for the full prompt on every new token would be wasteful, since most of that earlier context hasn't changed. So implementations cache the intermediate representations (the keys and values from note 04's attention mechanism) for tokens already processed, and reuse them instead of recomputing — this is why a long prompt costs more the first time you send it than the second. Prompt caching checks whether a matching prefix was already processed on a recent request; if not, "it processes the full prompt and caches the prefix once the response begins." The saving comes from a concrete place: "cached prefixes reuse the prior computation from when the prefix was first processed. The model doesn't need to re-process these tokens—it simply retrieves the cached intermediate representations. This elimination of redundant computation is why cache reads cost only 10% of the base input token price." Caches aren't permanent either — by default a cache entry has "a 5-minute lifetime" before it expires.

Lost in the middle

A bigger window is not the same thing as reliable recall across that window. A widely-cited study found accuracy depends heavily on where in the context a fact sits: "performance is often highest when relevant information occurs at the beginning or end of the input context, and significantly degrades when models must access relevant information in the middle of long contexts, even for explicitly long-context models." The paper's own framing: "we find that performance can degrade significantly when changing the position of relevant information, indicating that current language models do not robustly make use of information in long input contexts." A model given a 50-page document and asked about a fact on page 25 is measurably more likely to miss it than the same fact on page 1 or page 50 — and this held even for models built specifically for long context. Anthropic's own engineering guidance names a related, broader version of this: "as the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases," a phenomenon they call "context rot," which "affects all models" to varying degrees. A 128k-context spec tells you the ceiling, not the quality of attention across it.

Pro vs. amateur

Amateurs think of the context window as room for the prompt alone. Pros know the output has to fit inside the same budget — the window "holds the conversation history plus the new output Claude generates." Max out the input with a giant pasted document and there may be little or no room left for the model to actually answer; a maxed-out prompt is quietly starving its own reply of space. Amateurs put critical instructions once, at the top, and assume that's enough; pros know instruction strength isn't uniform across a long context — the lost-in-the-middle effect isn't only about retrieving facts from documents, it applies to instructions buried early in a long conversation too, so restating a real constraint near the end of a long context is a hedge against it getting diluted. Amateurs treat pasting the whole document in as the free option; pros know context is priced and quality-limited, so retrieval is often cheaper and better — Anthropic's own agent guidance recommends the opposite of front-loading: rather than pre-processing all relevant data up front, "maintain lightweight identifiers (file paths, stored queries, web links, etc.) and use these references to dynamically load data into context at runtime using tools." That's the retrieval (RAG) pattern in miniature, covered properly in Module 2 — but it exists because context is a finite, metered, imperfect resource, not a bottomless notebook. Amateurs read a 128k-context spec as 128k tokens of equally good attention; pros know the number on the box is a ceiling, not a quality guarantee — context rot and lost-in-the-middle are properties of every model tested so far, long-context ones included.
