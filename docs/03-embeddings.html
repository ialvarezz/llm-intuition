Embeddings

Before a model can compute with a token, the token becomes a vector — a point in a space where distance means similarity. That one idea powers everything from attention to RAG.

From symbols to geometry

A tokenizer (note 02) turns text into a sequence of integer IDs, but an ID by itself carries no meaning a neural network can compute with — token 4187 isn't closer to token 9021 just because 4187 is a smaller number. Before a transformer does anything useful with a token, that ID gets mapped to an embedding: a learned list of floating-point numbers — a vector — that positions the token as a point in a high-dimensional space. Google's ML crash course frames the problem an embedding solves: a one-hot vector means "large input vectors mean a huge number of weights for a neural network," and worse, "one-hot encoding vectors lack meaningful relationships, failing to capture semantic similarities between items". An embedding fixes both at once — it's built as "dense vector representations that capture semantic relationships and reduce the dimensionality of data" instead of throwing that structure away. Nobody hand-designs the numbers — the vector for each token is a set of weights the model learns during training, the same way it learns everything else. The word2vec paper that popularized this approach describes the goal as "novel model architectures for computing continuous vector representations of words", judged by how well those vectors do on "measuring syntactic and semantic word similarities". Words that show up in similar contexts end up with similar vectors, not because a human labeled them similar, but because training pulled them together.

Distance is meaning

Once tokens are points in space, similar meaning becomes nearby points — a fuzzy linguistic question turns into arithmetic. OpenAI's embeddings guide puts it plainly: for two embedding vectors, "small distances suggest high relatedness and large distances suggest low relatedness." The usual way to measure that distance is cosine similarity — the angle between two vectors, ignoring their length, asking only whether they point the same general direction. OpenAI recommends it because it's "computationally efficient for normalized embeddings and produces identical rankings to Euclidean distance"; Anthropic's docs make the same point about Voyage AI's embeddings: "Cosine similarity is equivalent to dot-product similarity... Cosine similarity and Euclidean distance will result in the identical rankings." The famous demonstration that this geometry captures more than same topic is vector arithmetic: take the vector for king, subtract man, add woman, and the nearest point is queen — commonly written king - man + woman = queen, encoding a relationship like male royal → female royal as a consistent direction that transfers across words. It's a striking result and an easy one to over-trust: the same mechanism also encodes whatever biases sit in the training data, producing stereotypes like doctor - man + woman = nurse. Treat the arithmetic as a vivid illustration of what training pulls into the geometry, not proof that meaning is literally linear algebra.

Token vs. document embeddings

Two different things get called embeddings. Token embeddings live inside the model — every vocabulary entry has a learned vector, stored in an embedding table that's part of the model's weights, the very first step a transformer performs on input before attention (note 04) runs. Sentence or document embeddings come from a dedicated embedding model, called through an API, that takes an arbitrary chunk of text and returns one fixed-length vector for the whole thing — what OpenAI's guide describes as "a vector (list) of floating point numbers" useful for "search with relevance ranking" among other tasks like clustering and anomaly detection. Anthropic's docs open with the same framing at the text level: "Text embeddings are numerical representations of text that enable measuring semantic similarity." Same underlying idea, two granularities: one number per token deep inside a model's forward pass, one number per chunk of text from a model you call directly. That granularity split matters for the roadmap — semantic search and retrieval-augmented generation (RAG), covered in Module 2, run entirely on document-embedding proximity: embed a corpus once, embed a query at request time, and return whichever stored documents land nearest.

Pro vs. amateur

Amateurs assume an LLM has one kind of intelligence at work; pros know embedding models and chat models are separate products, priced and chosen separately — Anthropic ships no embedding model of its own ("Anthropic does not offer its own embedding model") and instead directs developers to a specialist vendor, saying "you should assess a variety of embeddings vendors to find the best fit for your specific use case". Amateurs mix vectors from different sources in one index; pros never compare embeddings across models, because distance is only meaningful between vectors the same model produced — a 1024-dimension vector and a 3072-dimension vector aren't even comparable as numbers. Amateurs treat semantic similarity as objective; pros know it's a trained property of a specific model — a code-tuned embedding model and a general-purpose one rank the same pair of snippets differently, because what counts as similar was learned from different data. Amateurs treat higher dimensionality as strictly better; pros treat it as a cost/nuance knob — more dimensions capture finer distinctions but cost storage, memory, and search latency at scale, which is why production embedding models expose a dimensions parameter instead of shipping one fixed size.
