Tokens & Tokenization

A model never sees words. It sees tokens — subword fragments from a fixed vocabulary, learned from data, that stand between your text and everything the network does with it.

Why subwords

An LLM never sees words, sentences, or characters. It sees tokens — "the smallest individual units of a language model," which "can correspond to words, subwords, characters, or even bytes" depending on the tokenizer. One token per character would work, but sequences get too long: a paragraph that's 40 tokens under a subword scheme can be 200+ characters under a character scheme, and both context length and compute scale with sequence length (note 05). One token per whole word has the opposite problem: vocabulary explosion. A fixed vocabulary can't hold every word form, code identifier, and non-English string a user might type, and the paper that introduced subword segmentation for translation frames the mismatch directly: "Neural machine translation (NMT) models typically operate with a fixed vocabulary, but translation is an open-vocabulary problem." Subwords are the middle ground — fragments smaller than most words, larger than single characters, drawn from a fixed vocabulary typically in the 50,000–200,000 range, with individual bytes as the always-available fallback for anything the vocabulary has never seen.

How BPE carves text

The dominant algorithm is byte pair encoding (BPE). A real byte-level BPE tokenizer starts from raw bytes — "tokens initially go from 0 to 255... because we start with UTF-8 encoded bytes, and each byte can hold values from 0 to 255" — which guarantees any string in any language can always be represented. From there it's a purely statistical loop: count how often every pair of adjacent tokens occurs across a huge training corpus, merge whichever pair is most frequent into one new token, and repeat — "iteratively identify pairs... that appear the most often, define new tokens based on those, and replace them in the string... repeatedly merges the most frequent adjacent pair" — until the vocabulary hits its target size. Nobody hand-writes the rule that "ing" should be a token; it earns a slot because the merge process finds that byte sequence frequent enough. Production tokenizers add one more step before merging: GPT-2's tokenizer "applies a preprocessing chunking step using regex that segments text and runs BPE only within chunks to prevent undesirable merges such as joining words to punctuation."

One text, many tokenizations

Because the vocabulary and merges are learned from a specific training corpus, different model families end up with different tokenizers — and the same string can tokenize to a different number of tokens depending on which one reads it. This is measurable, not hypothetical: Anthropic's docs state that a newer tokenizer "produces approximately 30% more tokens than on earlier models" for identical input text, and warn against "reusing counts measured against earlier models" when estimating cost or context fit. Special tokens ride alongside the ordinary vocabulary too — BOS/EOS markers that flag where a sequence starts and ends, and chat-format markers that tell a fine-tuned model where a user's turn ends and the assistant's turn begins (note 01 covers that fine-tuning step). None of these are produced by BPE merges on user text; they're reserved IDs inserted at fixed points.

Pro vs. amateur

Amateurs budget prompts in words; pros budget in tokens, because context windows and pricing are both denominated in tokens, and the standard rule of thumb — "1 token is approximately 4 characters or 0.75 words for English text" — quietly breaks down for code and non-Latin scripts. Amateurs treat a token count as a fixed property of a string; pros know it's a property of (string, tokenizer) together, since the same text can produce 30% more tokens on one model's tokenizer than another's. Amateurs are baffled when a model mangles a username or a long number; pros recognize a rare string that fragmented into unfamiliar byte-level pieces. Amateurs treat leading whitespace as cosmetic; pros know " hello" and "hello" are frequently different tokens entirely, which matters the moment you're concatenating prompt strings by hand. And amateurs expect reliable spelling or letter-counting from a model; pros know it never had character-level access to begin with — it can only approximate an answer statistically, not iterate over letters the way code would.
