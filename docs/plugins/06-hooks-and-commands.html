Hooks & Commands

Prompting a model to 'always do X' is a suggestion. A hook makes it a guarantee — deterministic code on the agent loop's lifecycle events.

FIVE PLACES TO BOLT DETERMINISTIC CODE ONTO THE LOOP
Event names from Claude Code; other harnesses expose similar lifecycles.

// Suggestions vs. guarantees

Skills, MCP servers, and agent files all work the same way: put more information in front of the model and let it decide what to do with that information. A hook is different — it's deterministic code the harness itself runs at a fixed point in the agent loop, with no model judgment involved. Hooks can be shell commands, HTTP calls, MCP tool calls, or even single-turn or multi-turn LLM evaluations, but the common case is plain shell — because the whole point is removing judgment from the step, not adding another layer of it.

// The event catalog

Claude Code exposes a large, named catalog of lifecycle points — per-session events like SessionStart and SessionEnd, per-turn events like UserPromptSubmit and Stop, agentic-loop events like PreToolUse and PostToolUse, plus subagent, file, config, and compaction events. Each one fires at a documented, specific point: PreToolUse fires before a tool call executes, PostToolUse fires after it succeeds, Stop fires when the model finishes responding. That trigger point is what tells you whether a hook can still change the outcome — by the time PostToolUse runs, the tool has already executed.

// Block, allow, inject

Not every event can do the same things. Every event can inspect — read JSON describing what's happening. A subset can block — PreToolUse, UserPromptSubmit, and Stop among them — by exiting 2 or returning a JSON decision field. A different subset can inject context back into the conversation, and PreToolUse alone can rewrite the tool call's input before it runs. The contract is exit-code based: exit 0 means no objection and flow continues normally; exit 2 blocks the action and shows the hook's stderr to the model as feedback. Anything else lets the action proceed but surfaces an error notice. On top of the exit code, a hook can print structured JSON with a permissionDecision of allow, deny, ask, or defer for finer control than a bare exit code gives you.

Classic uses fall straight out of that contract: run the formatter after every edit is a PostToolUse hook; block writes to a protected path is a PreToolUse hook that inspects the tool input and denies the match; lint before commit is the same shape wired to a different matcher. Limits worth knowing: hooks can't trigger slash commands or other tool calls directly, a PostToolUse hook can't undo a tool that already ran, and if several PreToolUse hooks all try to rewrite the same input, only the last one to finish wins — non-deterministically.

// Slash commands

A slash command is invoked by the user, not the model — you type /name and the harness expands it into a prompt. In Claude Code today this has merged into skills: a skill file (SKILL.md) and the legacy .claude/commands/*.md form both create the same /name invocation and work identically. The old dedicated "slash commands" docs page now redirects to the skills docs — the mechanism didn't disappear, it got folded into the broader skill spec, which still documents the legacy commands/ form alongside it. A skill body supports argument interpolation for whatever text follows the command name — the full argument string, a specific positional argument, or a named argument declared in frontmatter.

The real distinction isn't "commands vs. skills" as two systems — it's user pulls vs. model decides, both routed through the same file. Frontmatter controls which side is on: disable-model-invocation: true means only you can invoke it, no auto-triggering; user-invocable: false means only the model can invoke it, hidden from the / menu. Left at defaults, a skill does both — you can type /name, and the model can auto-invoke it when the description matches the conversation.

// Pro vs. amateur

"Please always run the tests" in a prompt is a wish — it's read by the model and weighed against everything else competing for its attention that turn. A PostToolUse or Stop hook is a law: it runs every time, unconditionally, whether or not the model was thinking about it. Hooks also run with your own shell's permissions, which cuts both ways — that's what makes a formatter hook trivial to write and what makes an unaudited plugin hook dangerous; review a plugin's bundled hooks before installing it, the same way you'd review any script with your access. Keep hooks fast, too — they're synchronous with the loop, and a slow PreToolUse hook blocks the tool call it's guarding rather than running in the background. And reach for a hook only when the rule is mechanical — "run the formatter," "reject edits to a protected path" — not when it needs judgment; judgment calls are what skills are for.
