MCP Servers

The Model Context Protocol is why one integration works in Claude, Copilot, and everything else — a universal port between models and the outside world.

The N×M problem

Every AI assistant that wants to reach an external tool used to need its own bespoke integration — a Claude adapter for GitHub, a different Copilot adapter for GitHub, another for Slack, another again. Wire up N assistants against M tools by hand and you're maintaining N×M brittle, one-off connectors. MCP replaces that with one protocol in the middle: "[t]hink of MCP like a USB-C port for AI applications" — a standardized way to connect AI applications to external systems, the same way USB-C standardizes how devices plug into each other. Build a server once, against the protocol, and it works with any client that also speaks it — that's already true of the ecosystem: Claude, ChatGPT, VS Code, Cursor, and more all support MCP.

Tools, resources, prompts

The protocol defines a small, fixed vocabulary of what a server can offer: tools — functions the model itself decides to call, the same way any tool use works; resources — readable data (a file, a database row, a ticket) that the user or the model can pull into context, surfaced in Claude Code as @server:protocol://resource/path mentions; and prompts — templated workflows the user chooses to run, exposed as slash commands like /mcp__servername__promptname. The three-way split keeps the control boundary honest: the model decides tools, the user decides prompts, and resources just sit there as data for either to use. Three roles sit around this exchange — the host app (Claude Code, VS Code), a client connector inside it per server, and the server itself, all trading JSON-RPC 2.0 messages over a stateful, capability-negotiated connection.

Transports and config

A server is reachable one of two ways. stdio spawns a local process and talks over standard input/output — no network — the fit for anything needing local filesystem access. HTTP (the spec calls it streamable-http) reaches a remote endpoint over the network like any web API; a WebSocket variant exists for servers that push events unprompted rather than just answer requests. Both harnesses in this module configure servers as JSON, with one field-name gotcha: Claude Code keys its .mcp.json under mcpServers, while VS Code's .vscode/mcp.json keys the identical idea under servers. A minimal Claude Code entry:

{
  "mcpServers": {
    "shared-server": { "command": "/path/to/server", "args": [], "env": {} }
  }
}

and the VS Code equivalent:

{
  "servers": {
    "server-name": { "command": "npx", "args": ["-y", "@microsoft/mcp-server-playwright"] }
  }
}

The interoperability layer

Every other concept in this module — plugins, skills, agents, hooks — is scoped to one harness's own extension model. MCP is the piece explicitly designed not to be: a server written once against the protocol is the same process or endpoint whether Claude Code, Claude Desktop, or VS Code's Copilot Chat is connecting to it. Each harness's config file just points at it — that's the whole payoff of building once and integrating everywhere. Actually building a server is a task for the official SDKs, out of scope here beyond knowing they exist.

Pro vs. amateur

Amateurs connect every server that looks useful; pros connect what the task needs — a connected server's tool descriptions compete for context-window space by default, whether or not the model ever calls them, which is exactly why Claude Code built a whole tool-search system to defer them. Amateurs treat tool results like any other text; pros treat them as untrusted input — a result is data from outside the conversation, and nothing stops it from containing instructions phrased to look like the user's. Amateurs write a server before checking if one exists; pros search first — a working server for a mainstream tool is usually an install command away, not a build task. And amateurs treat a local stdio server as a black box; pros know it's just a process running with their own permissions on their own machine — read what you install.
