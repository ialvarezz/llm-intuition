Claude ↔ Copilot Compatibility

One integration, two harnesses. What carries over as-is, what needs translating, and what stays vendor-locked — component by component.

The portable layers

Portability tracks how open the underlying spec is, not which vendor built the harness. MCP is an open protocol; Agent Skills is an open standard; AGENTS.md is a shared file both harnesses read. Those three layers move between Claude Code and Copilot with little or no translation. An MCP server is written once and registered twice — Claude Code reads a mcpServers block in .mcp.json, VS Code Copilot reads a servers block in .vscode/mcp.json, and the server binary and protocol are identical. Skills follow the same Agent Skills spec on both sides: a SKILL.md with name and description frontmatter. GitHub documents full support across the Copilot cloud agent, code review, the CLI, and agent mode in VS Code and JetBrains; Claude Code implements the same spec and extends it with extra control fields — a base-spec skill moves with no changes, a skill leaning on Claude-only fields needs those stripped.

Instructions: AGENTS.md

AGENTS.md is the vendor-neutral instructions file — an open format governed under the Linux Foundation, used by tens of thousands of projects, with both Anthropic and GitHub Copilot on its supporter list. On the Copilot side the nearest AGENTS.md in the directory tree takes precedence, and a lone root CLAUDE.md is even picked up as a fallback. The practical move: keep one canonical file and have the other include it — a one-line CLAUDE.md pointing at AGENTS.md beats two hand-maintained copies that drift.

Agents: same idea, different files

A Claude Code subagent and a Copilot custom agent are the same concept — a system prompt plus a tool allowlist plus a model choice, scoped to a role. The file formats differ (subagent markdown vs. .agent.md in .github/agents), and Claude Code's version carries extras — persistent memory, worktree isolation, per-subagent hooks — that have no documented Copilot counterpart. Translating the shared fields is mechanical: copy the prompt body, map tools to tools, model to model. The extras are lossy.

The vendor-locked layer

Hooks are the least portable component. Claude Code publishes a full lifecycle catalog — dozens of events, five execution types, a structured JSON contract. Copilot documents a hooks feature in one line: run your own commands at key points in the agent's loop. The concept crosses over; the granular event names don't. Slash commands fare better: Claude Code's command templates (now part of the skills system) map one-for-one onto Copilot's Prompt Files (*.prompt.md) — same role, different file extension.

A portability strategy

Put as much of an integration's knowledge as possible into the portable layers — one MCP server, one AGENTS.md, skills written to the base Agent Skills spec — and keep the harness-specific layer as a thin wrapper on top. A team that does this can switch assistants, or run both side by side, and only the wrapper needs rewriting.

Pro vs. amateur

Amateurs design for one harness and port later; pros design for the portable layers first and treat vendor-specific glue as the last thing they add. Amateurs copy the same standards into CLAUDE.md and AGENTS.md; pros pick one canonical file and include it from the other — duplicates drift the moment one is edited. Amateurs ship compatibility claims from a single search result; pros verify against current docs — this layer shifts quarterly, and this page's matrix is dated to its research fetch. Amateurs assume cross-tool means rewrite; pros standardize on MCP + AGENTS.md + base-spec skills and switch assistants with near-zero rework.
