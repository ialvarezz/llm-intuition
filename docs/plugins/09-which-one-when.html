Which One, When

Skill, MCP server, agent, hook, command, or just a prompt? Start from the failure mode you're fixing — the right artifact falls out.

START FROM THE FAILURE MODE, NOT THE TECHNOLOGY
A heuristic, not a law — real plugins usually combine two or three artifacts.

Start from the failure mode

Every artifact in this module fixes exactly one kind of failure, and the failure names the fix. "The model doesn't know how we do X" is a knowledge gap: create a skill "when you keep pasting the same instructions, checklist, or multi-step procedure into chat, or when a section of CLAUDE.md has grown into a procedure rather than a fact" [S1] — concept 04 covers writing one. "The model can't reach system X" is a capability gap, not a knowledge gap: no amount of instruction fixes a model with no path to a filesystem, API, or database; that's what tool use and, at scale, MCP servers exist to close (concept 03). "This sub-task pollutes/overflows the main context or needs different tools" is a context-management problem: Claude Code's context: fork mode is explicit that "The skill content becomes the prompt that drives the subagent. It won't have access to your conversation history." [S1], and long-horizon guidance names sub-agent architectures (delegated exploration returning condensed summaries) as a standard way to keep a large task's context tight [S4] — concept 05 covers the dispatch contract. "This rule must hold every time, no exceptions" is a reliability problem a skill can't solve, because the model can still choose to skip it: hooks "are user-defined shell commands that execute at specific points in Claude Code's lifecycle. They provide deterministic control over Claude Code's behavior, ensuring certain actions always happen rather than relying on the LLM to choose to run them" [S2] — concept 06 covers the event catalog. "I keep typing the same request" is a repetition problem: a slash command, also concept 06. "It's a one-off" doesn't earn an artifact at all — a plain prompt is cheapest for a task that won't recur.

The cost column

Each artifact sits at a different point on a cost curve, and the curve is why escalation order matters. A skill is markdown: "a skill's body loads only when it's used, so long reference material costs almost nothing until you need it" [S1] — the cheapest artifact that adds judgment. An MCP server is running software: a live dependency to install, authenticate, version, and patch (concept 03). An agent dispatch costs latency and tokens per call — "Agentic systems often trade latency and cost for better task performance, and you should consider when this tradeoff makes sense" [S3], and "the autonomous nature of agents means higher costs, and the potential for compounding errors" [S3]. A hook is a shell script on the hot path, running at "specific points in Claude Code's lifecycle" on every matching event [S2] — cheap per call, but now part of the critical path, and a broken hook can block work outright rather than just answer wrong.

A plugin usually isn't one artifact from this list — it's a bundle. A real plugin might ship a skill that teaches the procedure, a hook that guarantees a linter runs before every commit regardless of whether the model remembers, and a command that gives a human a fast manual trigger. The framework above answers "which artifact for which failure," not "which single artifact for this plugin" — most plugins answer several of the six questions at once, one artifact per failure mode they cover.

Every artifact installed is also something somebody now owns. Descriptions drift — the goal is "the _smallest_ _possible_ set of high-signal tokens that maximize the likelihood of some desired outcome" [S4], and a stale description misleads the model about when to invoke the thing, it doesn't just waste tokens. Docs-linked instructions rot the moment the thing they reference moves, with no error to signal it. And someone must own each artifact — hook, skill, or MCP server is infrastructure with the same lifecycle as code: a person has to notice when it breaks, update it, or retire it.

Escalate on evidence

Pros pick the cheapest artifact that fixes the failure mode, and escalate only on evidence: prompt → command → skill → agent/MCP. Each step up is only justified once the cheaper step has been tried and demonstrably fails — "you should consider adding complexity _only_ when it demonstrably improves outcomes" [S3]. They also measure the context cost of what they install: every tool description, injected instruction, and retrieved snippet sits in the same bounded context window and gets billed as input tokens whether the model uses it or not — the discipline is keeping that budget "informative, yet tight" [S4].

Pro vs. amateur

Amateurs pick the shiniest tool — an MCP server where a 30-line skill would do, running software with its dependencies, auth, and upkeep, to solve a problem markdown and a description would have solved cheaper. Pros pick the cheapest artifact that fixes the failure mode and escalate only on evidence. And pros delete plugins that stopped earning their tokens: an artifact that made sense when installed can stop paying for itself, and it keeps costing context and maintenance attention regardless — the same principle that justifies adding an artifact justifies cutting the ones that no longer earn their place.
