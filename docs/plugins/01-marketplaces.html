Plugin Marketplaces

A marketplace is just a catalog your AI harness can install from — a JSON file pointing at git repos. Understanding that demystifies the whole ecosystem.

Why extend the harness

Concept 00 framed a plugin as a change to the harness, not the model. The lazy alternative to a plugin is repeated prompting — pasting the same instructions into every session — and that's copy-paste engineering: it doesn't version, doesn't share, and doesn't get reviewed. Packaging those instructions, tools, and hooks turns them into an artifact with a name and a source someone else can read before running. A marketplace is how that artifact gets found.

A catalog, not a store

A marketplace is a git repository, URL, or local path containing a .claude-plugin/marketplace.json file that lists plugins and where to find them — it provides "centralized discovery, version tracking, automatic updates, and support for multiple source types." Each plugin entry needs a name and a source; that source can be a relative path, a github repo, a url, a git-subdir, or an npm package. The marketplace itself hosts none of it — it's a pointer, and the plugin's actual code can live in a completely different repository owned by someone else entirely.

Install flow

Two steps, matching that catalog/content split. Register a catalog with /plugin marketplace add <source> — that does nothing but make the list of plugins visible. Then install a specific plugin out of it with /plugin install <plugin-name>@<marketplace-name>. Installed plugins get copied to a local cache and can be toggled on and off, which matters because every enabled plugin's tool descriptions sit in the context window on every turn — the same context-budget point Concept 00 made about tools in general.

Trust: you are running their code

A marketplace listing doesn't mean the plugin was vetted — it means a pointer got added to a catalog. Installing a plugin puts someone else's instructions, and in the case of hooks or MCP servers someone else's executable code, inside your session with your access. VS Code makes this explicit with a trust dialog the first time you install a third-party publisher's extension; Claude Code's equivalent discipline is manual: review the source, pin a ref or sha instead of tracking a moving branch, and treat a new plugin the way you'd treat a new dependency.

Pro vs. amateur

Amateurs treat a marketplace as a store that hosts and vets plugins. Pros know it's metadata pointing at git — the source can live anywhere git reaches, and being listed implies nothing about review or safety. Amateurs install and forget; pros pin and review versions like any dependency, since omitting a version pin means every new commit becomes a new version silently. And amateurs judge a plugin by which marketplace it's in — official, community, or otherwise; pros know the marketplace a plugin ships in says nothing about its quality, and read the source. Private and team marketplaces, not the public ones, are where most real deployment happens: the same source types work against a private repo with token-based auth, pointed only at your own team.
