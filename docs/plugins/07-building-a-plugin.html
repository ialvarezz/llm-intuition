Building & Publishing

    STEP 1/6 — SAME PATH FOR EVERY PLUGIN, WHATEVER IT BUNDLES
    
    
    Abbreviated file contents — the note carries the full, copy-pasteable versions.
    
  
    

  
    
    
    // The worked example: pr-buddy
    
    
    The rest of this module has described plugin parts — a marketplace entry, a manifest field, a skill's frontmatter, an agent's tool list. This concept builds one complete, small plugin end to end: pr-buddy, bundling one skill (review-pr, a checklist-driven PR review) and one agent (test-writer, which writes missing tests). The note carries the full, copy-pasteable files; this page walks the structure.
    

    
    // Scaffold and manifest
    
    
    A plugin's manifest is required at .claude-plugin/plugin.json, and everything else — skills, commands, agents, hooks — has to sit at the plugin's root, not inside .claude-plugin/. The manifest only strictly requires name, a unique kebab-case identifier that also sets the skill namespace: a skill called hello in a plugin named my-first-plugin is invoked as /my-first-plugin:hello. pr-buddy's manifest adds description and version on top of the required minimum — description is what a user browsing the plugin manager sees, and version is what gates future updates.
    

    
    // The skill and the agent
    
    
    Concept 04 established that a skill's description is the whole interface — the model decides whether to load the body from that field alone, so it has to state both what the skill does and when to use it, not just its domain. review-pr's description names a concrete trigger phrase instead of a generic label. Concept 05 established that a subagent's tool allowlist is a hard wall, not a suggestion the model might drift from — test-writer's frontmatter restricts tools to exactly Read, Write, Bash, nothing more, so it cannot touch source files outside the one it was dispatched to cover.
    
    
    MCP servers can be bundled into a plugin too, via a .mcp.json at the plugin root or an inline field in plugin.json — see Concept 03 for how MCP fits in. pr-buddy doesn't need one.
    

    
    // Test locally, then publish
    
    
    Before pr-buddy touches a marketplace, claude --plugin-dir ./pr-buddy loads it directly from a local path, and /reload-plugins picks up edits without restarting the session. A real local test pass runs in a fresh session and checks three things: the skill triggers on a realistic phrase, the agent is dispatchable, and — the part amateurs skip — a near-miss phrase that shouldn't trigger the skill doesn't.
    
    
    Publishing has two moving parts: pr-buddy's own git repo, and a separate marketplace repo containing .claude-plugin/marketplace.json with a name, owner, and a plugins array pointing at pr-buddy's source. /plugin marketplace add you/your-marketplace followed by /plugin install pr-buddy@your-marketplace installs it from there. Once published, updates are gated by version, not by commits — pushing a fix without bumping version does nothing for anyone who already installed the prior version. Bumps follow semver: PATCH for a fix, MINOR for a new skill or agent, MAJOR for a breaking change to what's already there.
    

    
    // Pro vs. amateur
    
    
    Set version and start a changelog from day one — an unversioned plugin lets the commit SHA do the versioning silently, and a teammate's session can pick up a half-finished commit mid-edit. Test in a fresh session, not the one where the plugin was designed — that session already discussed the trigger phrase at length, which tells you nothing about whether a real user's request would have matched the description cold. Write the README for the human deciding whether to install pr-buddy; write the skill and agent descriptions for the model deciding whether to trigger them — one document trying to serve both jobs under-serves whichever reader shows up second. And start as a local --plugin-dir install, used in real PR reviews for a while, before it earns a marketplace entry — a week of real use surfaces the over-broad triggers a same-day publish never catches.
