AI Plugins

AI Plugins — from zero to shipping

  • Module A1 · 45–60 minutes
  • Marketplaces, plugins, MCP, skills, agents, hooks
Opening

Why extend your assistant at all

  • Repeated prompting is copy-paste engineering
  • It doesn't version, share, or survive a teammate joining
  • Packaging turns instructions into an artifact: named, versioned, reviewable
  • That artifact is a plugin — the subject of this whole deck
Opening

Agenda — nine sections

  • Primer · Marketplaces & plugins · MCP
  • Skills · Agents · Hooks & commands
  • Build & ship · Close (decision matrix)
Primer

Next-token prediction, looped

  • An LLM predicts the next token — that's the whole operation
  • Chat is that operation run in a loop over a growing transcript
  • Nothing about "conversation" requires new machinery
Primer

The context window is working memory

  • "The amount of text a model can look back on and reference"
  • Not the training corpus — a bounded working memory
  • Every plugin, tool, or injected context competes for this same space
Primer

Tool use — the model asks, the harness acts

  • A raw LLM has no hands: can't run a command or call an API
  • Model returns a structured call; the surrounding app executes it
  • Model decides per-turn whether a request maps to a tool
Primer

The agent loop

  • Workflows: predefined code paths. Agents: the model directs itself
  • Loop: model decides → harness acts → result re-enters context → repeat
  • Same next-token loop as chat, just fed tool output each turn
Primer — thesis

Plugins extend the harness, not the model

  • The model's weights never change between tool calls or plugins
  • The harness is what varies: Claude Code, Copilot, etc.
  • A plugin is a new tool, new context, or new command the harness gains
Marketplaces & plugins

A catalog, not a store

  • A marketplace is metadata — a list of pointers, not hosted code
  • "Centralized discovery, version tracking, automatic updates"
  • Backed by git repos, URLs, or local paths
Marketplaces & plugins

marketplace.json anatomy

{
  "name": "your-marketplace",
  "owner": { "name": "Your Name" },
  "plugins": [
    { "name": "pr-buddy", "source": "./plugins/pr-buddy" }
  ]
}
  • Required: name · owner · plugins. Each entry: name · source
Marketplaces & plugins

Install flow — a two-step handshake

/plugin marketplace add user-or-org/repo-name
/plugin install plugin-name@marketplace-name
  • Step 1 registers the catalog. Step 2 installs by name from it
Marketplaces & plugins

Plugin = the container

enterprise-plugin/
├── .claude-plugin/
│   └── plugin.json
├── skills/
├── commands/
├── agents/
├── hooks/
├── bin/
└── .mcp.json
  • Any mix of skills, agents, hooks, MCP servers, LSP servers
Marketplaces & plugins

The manifest

  • .claude-plugin/plugin.json — only name required
  • Sets the namespace: my-plugin:hello, not /hello
  • Skip version and every commit silently becomes a new release
Marketplaces & plugins

Trust — you are running their code

  • A marketplace listing doesn't vet or sandbox anything for you
  • Hooks and MCP servers run with your session's own access
  • Pin to a ref/sha; review source before installing, like any dependency
MCP

The N×M problem

  • N assistants × M tools = N×M bespoke, brittle integrations
  • MCP: "a USB-C port for AI applications"
  • Build one server against the protocol; every compliant client can use it
MCP

Tools, resources, prompts

  • Tools — functions the model decides to call
  • Resources — context/data for user or model to pull in
  • Prompts — templated workflows the user chooses to run
MCP

The handshake

  • Host (Claude Code) ⟶ Client (one connector per server) ⟶ Server
  • JSON-RPC 2.0, stateful, capability negotiation on connect
  • Servers can add tools mid-session via list_changed notifications
MCP

One server, both harnesses

// Claude Code — .mcp.json
{ "mcpServers": {
  "github": { "command": "npx", "args": ["-y","@some/mcp-server"] } } }

// VS Code Copilot — .vscode/mcp.json
{ "servers": {
  "github": { "type": "http", "url": "https://api.githubcopilot.com/mcp" } } }
  • Same server, same protocol — just a different top-level key
Skills

What a skill is

---
name: my-skill-name
description: A clear description of what this skill does and when to use it
---
# My Skill Name
[Instructions Claude will follow when this skill is active]
  • Prose the model reads — not code the harness executes
Skills

Progressive disclosure — three levels

  • Level 1: name + description, loaded always, at startup
  • Level 2: full SKILL.md body, loaded when relevant
  • Level 3+: bundled files, loaded only as needed, via filesystem
Skills

The token math

  • Level 1 metadata: ~100 tokens per skill
  • Level 2 body: under 5k tokens
  • Level 3+ resources: effectively unlimited — never read as text
Skills

The description is the API

  • Model decides whether to load the body from the description alone
  • A body that never triggers delivers zero value
  • Needs both what the skill does and when to use it
Skills

Good vs. bad description

bad:  "powerful toolkit for all your document needs"

good: "Use when reviewing a pull request or when asked
       to check a diff before merge."
Skills

Writing the body

  • Heading + Instructions (step-by-step) + Examples
  • Keep SKILL.md under 500 lines — split unwieldy files out
  • Rendered body enters context once and stays for the session
Skills

Testing — fresh session, negative case

  • Measure two axes: does it trigger, and is the output right
  • Run each prompt fresh, with and without the skill enabled
  • Test near-miss prompts too — over-triggering is the amateur miss
Skills

Failure modes

  • Vague description → never triggers
  • Over-broad description → triggers everywhere
  • Giant body → context bloat, persists all session
  • Contradicts CLAUDE.md → the model resolves confusion mid-task
Agents

The agent file

---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: sonnet
---
You are a code reviewer. When invoked, analyze the code and
provide specific, actionable feedback.
Agents

Why delegate

  • Fresh context — the parent transcript never enters the subagent
  • Restricted tool sets are a hard wall, not a suggestion
  • Cheaper models for mechanical, search-and-report work
Agents

The dispatch contract

  • Subagent gets its task message once — no mid-task follow-up
  • It works alone and returns exactly one final message
  • A result lands in the orchestrator's context, not the user's view
Agents

The work-order rule

bad:  "fix the bug"

good: "Bug: X fails on empty input in file Y, line Z.
       Deliverable: a fix plus one test covering it.
       Constraint: don't touch the public API."
Agents

Orchestration patterns

  • Fan-out: independent work in parallel, findings synthesized
  • Pipeline: sequential, each stage passes context to the next
  • Nested subagents: a delegated task splits further, one layer deeper
Agents

Model-per-agent economics

  • model is set per subagent, independent of the parent
  • Fast/cheap model for mechanical work; frontier model for judgment
  • Default is inherit — an unset agent silently costs more or less
Agents

When NOT to delegate

  • Task leans on context already in the current conversation
  • Cheaper to do inline than compress into a self-contained dispatch
  • "Start simple, add complexity only when it measurably improves"
Hooks & commands

Suggestions vs. guarantees

  • Skills and MCP put information in front of the model — it decides
  • Hooks are deterministic code the harness runs, no model judgment
  • "Run the tests" as a prompt is a wish; as a hook, it's a law
Hooks & commands

The event catalog

  • 27 distinct lifecycle events — session, per-turn, agentic-loop
  • PreToolUse can block or rewrite input, before the tool runs
  • PostToolUse can inject context or rewrite output — but can't undo
Hooks & commands

Slash commands, in one slide

  • User types /name; harness expands it into a prompt
  • Now merged into skills — one file format, two invocation modes
  • disable-model-invocation vs. user-invocable: false

Build & ship

The skill + the agent

review-pr: "Use when reviewing a pull request or
             asked to check a diff before merge."

test-writer:
  tools: Read, Write, Bash
  "Writes missing tests for a given file.
   Do not modify the file under test."
Build & ship

Test locally, negative case included

$ claude --plugin-dir ./pr-buddy
> "review this PR"                 ✓ skill triggered
> dispatch test-writer on file X   ✓ agent writes test
> "fix this typo"                  ✓ negative: no trigger
Build & ship

Publish

git push origin main
> /plugin marketplace add you/your-marketplace
> /plugin install pr-buddy@your-marketplace
  • Bump version deliberately — pushing commits alone isn't enough
Build & ship

Install it in Copilot too

  • MCP and Agent Skills are open standards — near-zero rework
  • AGENTS.md is the shared, vendor-neutral instructions file
  • Hooks and vendor-specific frontmatter are the least portable layer
Close

The decision matrix

  • Knowledge gap → skill. Capability gap → MCP server
  • Context pollution → subagent. Must-hold-every-time → hook
  • Repetition → slash command. One-off → plain prompt
Close

Escalate only on evidence

  • prompt → command → skill → agent/MCP
  • Each step up is justified only once the cheaper step demonstrably fails
  • Delete plugins that stopped earning their tokens
