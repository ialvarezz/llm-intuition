Plugin marketplaces

Plugin marketplaces — anatomy of an ecosystem

  • ~45 minutes + open Q&A
  • What a marketplace is · what's inside a plugin · when to use which part · good practices
  • Running example throughout: a code-review plugin called review-buddy
Why this matters

Stop re-pasting the same prompt

  • Everyone keeps a private stash of prompts — copied, drifted, unversioned
  • A plugin turns that stash into a named, versioned, reviewable artifact
  • A marketplace is how a team shares those artifacts
  • Promise: by the end, you can read any plugin like a datasheet
1 · Marketplace

A catalog, not a store

  • A marketplace is one metadata file — a list of pointers, no hosted code
  • Plugins themselves live in git repos, URLs, or local folders
  • It buys you discovery, version tracking, and updates
{
  "name": "team-marketplace",
  "owner": { "name": "Platform Team" },
  "plugins": [
    { "name": "review-buddy",
      "source": "https://git.example.com/tools/review-buddy",
      "version": "1.2.0" }
  ]
}
1 · Marketplace

A plugin is a folder — open it up

review-buddy/
├ plugin.json
The nameplate: name, version, description. The only required file.
├ commands/
Entry points a human types — /review expands to a prompt template.
├ skills/
Packaged expertise the assistant loads on demand — the team's review checklist.
├ agents/
Sub-assistants for delegated jobs — the reviewer that reads big diffs.
├ hooks/
Automatic triggers on events — run lint before any commit lands.
└ .mcp.json
Tool connections to external systems — fetch diffs from the repo host.
  • Types you'll meet: single-purpose (one command), toolkit (many components), pure-config (just connections)
1 · Marketplace

How consumption works

1 · Add the marketplace. Your harness fetches the catalog file. Metadata only.
2 · Browse. You search names and descriptions. Still nothing downloaded.
3 · Install. The plugin's files are copied to your machine.
4 · Nothing runs yet. It's text files on disk. You can open and read every one.
5 · Session start. The harness loads commands, skills, and agents into the assistant's context, and starts any MCP servers.
6 · Updates. A version bump in the catalog is how you're offered the new release.
2 · Components

Mental model: harness vs model

  • The model: fixed weights that predict text. It never changes when you install anything.
  • The harness: the application around it — executes tools, injects context, enforces rules
  • Every plugin component extends the harness, never the model
  • Keep this; it answers most "but how does it learn…" questions — it doesn't
2 · Components

Commands — the entry point

  • A human types it; it expands into a prompt template — nothing more
  • Cheapest component: deterministic start, zero standing cost
commands/review.md
---
description: Review a change against the team checklist
---
Review the current diff.
Use the review-checklist skill.
For diffs over 300 lines, dispatch the reviewer agent.
Report findings ranked by severity, with file:line refs.
2 · Components

Skills — packaged expertise

  • A markdown file of know-how the assistant pulls in when relevant
  • Only the one-line description is always loaded; the body loads on demand — context stays lean
  • Contrast with commands: a human invokes a command; the model reaches for a skill
skills/review-checklist/SKILL.md
---
name: review-checklist
description: Team checklist for reviewing any code change
---
1. Does the change do what the description claims?
2. Error paths: what happens on bad input?
3. Naming matches surrounding code …
2 · Components

MCP servers — the hands

  • A standard protocol for giving the assistant tools that touch real systems
  • review-buddy uses one to fetch_diff and post_comment on the repo host
  • The only component that is running code, not text — remember that at install time
assistant ──(tool call)──▶ MCP server ──(API)──▶ repo host
          ◀──(result)────           ◀──(data)───
2 · Components

Hooks — the reflexes

  • Event → shell command. Deterministic. No model judgment involved.
  • review-buddy: before any commit, run the linter; block on failure
  • Every other component advises the model; a hook constrains it
hooks/hooks.json
{ "PreCommit": [ { "command": "lint --strict",
                   "blocking": true } ] }
2 · Components

Agents — the delegates · all five in one run

  • A sub-assistant with its own instructions, own context, limited tools — spawned for a bounded job, reports back
1 · Human: /review — the command expands into the instruction prompt.
2 · Spawn: the diff is 800 lines, so the assistant dispatches the agent — your session stays clean.
3 · Fetch: the agent calls the MCP server's fetch_diff tool.
4 · Judge: the skill's checklist guides what the agent looks for.
5 · Report: findings come back ranked; you approve a fix.
6 · Gate: the fix is committed — the hook runs lint before it lands.
3 · When to use what

Pick by the job, not the buzzword

I want to run a repeatable task on demand…
Command. A human-triggered prompt template. Start here — it's the cheapest thing that works.
I want to teach it my team's way of doing things…
Skill. Knowledge gap → write it down once; the model loads it when relevant.
I want to connect an external system…
MCP server. Capability gap → the assistant can't reach what it has no tool for.
I want a rule enforced every time, no exceptions…
Hook. If "usually" isn't good enough, don't ask the model — trigger on the event.
I want to delegate a bounded job without flooding my session…
Agent. Context pollution → give the job its own context and take back only the result.
3 · When to use what

Why review-buddy is built this way

  • Checklist as a command instead of a skill? It would bloat every review prompt — skills load on demand
  • Lint as a skill instead of a hook? The model might skip it — a hook can't be skipped
  • Pasting diffs instead of MCP? Stale and manual — the tool fetches live state
  • Reviewing inline instead of an agent? An 800-line diff floods the session that asked for it
  • Each component earns its place by what breaks without it
4 · Good practices

BKMs — building

  • Start with a command; add components only when the cheaper one demonstrably fails
  • Escalation ladder: plain prompt → command → skill → agent / MCP
  • Keep skills short — every loaded token competes with the actual task
  • One plugin = one job. Split toolkits that try to do everything
  • Bump the version deliberately — pushing commits alone updates nobody
4 · Good practices

BKMs — consuming & trust

  • Read before you install — plugins are text files, so audit them like a code review
  • Exception: MCP servers run code — treat them like any third-party dependency
  • Pin versions for anything your team relies on
  • Context is a budget: every installed plugin competes for the same working memory — uninstall what stopped earning its place
