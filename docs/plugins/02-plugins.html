Plugins: The Container

A plugin is a directory with a manifest. Everything installable — skills, agents, hooks, MCP servers, commands — ships inside one. Learn the layout once and every plugin you open makes sense.

One unit, many components

Concept 01 covered the marketplace — the catalog that points at a plugin's source. This concept opens the thing being pointed at. A plugin is "a self-contained directory of components that extends Claude Code with custom functionality" — components include "skills, agents, hooks, MCP servers, LSP servers, and monitors." Whatever you'd otherwise paste into every session by hand — a tool connection, a specialized subagent, a hook, a shortcut command — goes in the container once, and the container installs as a unit. Components inside a plugin get namespaced (/my-plugin:hello instead of a bare /hello) so two plugins can each ship a skill with the same short name without colliding.

Anatomy on disk

The layout mirrors the component list directly: .claude-plugin/plugin.json is the manifest; skills/ holds skills as <name>/SKILL.md directories; commands/ holds skills as flat markdown files; agents/ holds custom agent definitions; hooks/ holds hooks.json; and .mcp.json at plugin root holds MCP server configuration. One hard rule: "Don't put commands/, agents/, skills/, or hooks/ inside the .claude-plugin/ directory. Only plugin.json goes inside .claude-plugin/." Nothing is mandatory beyond that — a plugin can be a single skill and a manifest and still be complete.

The manifest

plugin.json is what a marketplace reads before installing anything. Only name is required — a unique identifier that doubles as the namespace prefix. Everything else is metadata: description, version, author. Skip version and the harness falls back silently to the git commit SHA as the version, so every commit becomes an un-communicated release. Component-path fields like skills, commands, and agents can override the default folders, but they don't all behave the same way — most replace the default scan, while skills is the exception and always adds to it.

Install lifecycle

What loads and when splits along component lines. MCP servers "start automatically when plugin is enabled" — they're a running process. Skills and commands are "automatically discovered when plugin is installed", but a skill body only enters context when actually invoked. Hooks register their event matchers at load and fire later on the matching lifecycle event. For local development, claude --plugin-dir ./my-plugin loads a plugin from a path with no marketplace step, and /reload-plugins picks up edits without restarting.

Pro vs. amateur

Amateurs build a kitchen-sink plugin. Pros ship one good skill. Enabled plugins add to system-prompt context, which is why the system supports toggling them off when unneeded — ten mediocre skills cost that budget always, whether used or not. Amateurs pick generic component names. Pros name for discoverability. Namespacing (plugin:skill) already prevents collisions; a clear plugin name is what makes it findable in a crowded list. Amateurs publish straight to a marketplace. Pros test locally first. --plugin-dir plus /reload-plugins is the exact same load path without the shared-catalog risk. Amateurs treat plugin.json as a formality. Pros treat it like package.json. Semver and changelogs matter the moment teammates depend on a plugin — an unversioned manifest tracks a moving commit SHA instead of a deliberate release.
