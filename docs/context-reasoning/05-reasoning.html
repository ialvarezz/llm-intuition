Reasoning, in Depth

Reasoning isn't a separate faculty a model switches on. It's more tokens spent before the answer — and more tokens is a knob, not a guarantee.

Read 9 minLevel IntermediatePrereq Module 00, Concept 01

// More tokens, not a different process

An autoregressive model produces one token at a time, and each new token is computed from everything already in the context — including tokens the model itself just generated. That's the entire mechanical content of "reasoning" in a language model. There is no separate reasoning module, no scratch space consulted before deciding what to say, no cognitive faculty distinct from "predict the next token" that switches on for hard problems. The only way an autoregressive model gets more computation applied to a problem is by writing more tokens first — every additional token is another full forward pass, conditioned on everything written so far, including its own prior output. An intermediate token that states a sub-conclusion isn't decoration; it's compute the model would not otherwise get to spend, and it becomes part of the input to every token generated after it.

So reasoning is spending more generated tokens before committing to a final answer, not a distinct process bolted onto next-token prediction. Whether those tokens are visible chain-of-thought in the same response or routed into a separate "thinking" channel the API treats specially, the mechanism is identical — more forward passes, conditioned on the growing sequence, before the model outputs the tokens it treats as the answer. Which means everything already true about tokens elsewhere in this module is true of reasoning tokens specifically: they cost money, they take time to generate, and more of them isn't free.

// Chain-of-thought and extended thinking

Wei et al.'s 2022 paper made this practical rather than theoretical: prompting a model with a few worked examples that show intermediate steps, instead of examples that jump straight from question to answer, "significantly improves the ability of large language models to perform complex reasoning" [S1]. The reported gains aren't marginal — "prompting a 540B-parameter language model with just eight chain of thought exemplars achieves state of the art accuracy on the GSM8K benchmark of math word problems, surpassing even finetuned GPT-3 with a verifier" [S1]. A handful of few-shot examples showing steps outperformed a model fine-tuned for the task and paired with an external verifier. That gain isn't free: every intermediate step is tokens the model wouldn't otherwise generate, meaning added latency and added cost per call. Chain-of-thought is a trade — tokens for accuracy — and it pays off most on genuinely multi-step problems where a direct jump from question to answer gives the model no intermediate computation to work with.

Current model providers have since turned the same idea into a first-class API feature: a dedicated reasoning phase with its own token budget, structurally separated from the final answer but mechanically identical to it. Anthropic's extended thinking is representative — turning it on exposes the mechanism directly: "Claude creates thinking content blocks where it outputs its internal reasoning. Claude incorporates insights from this reasoning before crafting a final response" [S2]. The budget behaves like a compute knob: "Larger budgets can improve response quality by enabling more thorough analysis for complex problems, although Claude may not use the entire budget allocated, especially at ranges above 32k" [S2], and the returns diminish — "Higher token counts enable more comprehensive reasoning but with diminishing returns depending on the task" [S2]. The trace itself may be shown in full, summarized, or omitted, but the billing doesn't change: "you're charged for the full thinking tokens generated by the original request, not the summary tokens," so "the billed output token count will not match the count of tokens you see in the response" [S2]. A short or hidden trace tells you nothing about how much was actually spent producing it.

// A trace isn't proof

It's tempting to conclude that a longer, more detailed reasoning trace is evidence of a more reliable answer — if you can see the steps and they look sound, the conclusion must be sound too. Anthropic's faithfulness research directly undermines that. It investigates whether "the stated reasoning is a faithful explanation of the model's actual reasoning (i.e., its process for answering the question)" [S3], by perturbing the chain-of-thought — inserting mistakes, paraphrasing it — and watching whether the final answer moves. If the visible trace really were the computation path, corrupting it should reliably corrupt the answer. Instead: "Models show large variation across tasks in how strongly they condition on the CoT when predicting their answer, sometimes relying heavily on the CoT and other times primarily ignoring it" [S3]. A trace that reads as the reason for an answer can be a plausible-sounding narrative generated alongside an answer the model would have reached anyway — a post-hoc rationalization, not a report of what happened inside the forward passes that produced it.

The most counterintuitive finding sharpens this: "As models become larger and more capable, they produce less faithful reasoning on most tasks we study" [S3]. Faithfulness does not simply track capability upward — a more capable model is not automatically a more honest narrator of its own process. The paper also cautions against the too-simple mechanical story that more tokens is the whole explanation: "CoT's performance boost does not seem to come from CoT's added test-time compute alone" [S3]. The overall finding is a caveat, not a dismissal — "CoT can be faithful if the circumstances such as the model size and task are carefully chosen" [S3] — but a trace looking coherent and well-structured is not, by itself, proof it's an accurate account of how the answer was reached. Confident wrong reasoning is a real and common failure mode: fluent, step-numbered, still wrong, because nothing about the mechanism guarantees the steps are true, only that the model had more context to condition on, including its own possibly-wrong earlier steps. And when a model lacks the underlying fact a question needs, spending more tokens "reasoning" toward it doesn't manufacture that fact — it just produces a longer, equally fluent way of still being wrong.

// Pro vs. amateur

Amateurs treat a longer trace, or turning on extended thinking, as a free accuracy upgrade — more tokens, so a better answer, no further scrutiny needed. Pros treat the reasoning-token budget as a cost/quality knob with diminishing returns [S2], tuning it per task rather than maxing it out by default. Amateurs read a clean, step-numbered trace and take it as proof the answer is right; pros verify conclusions independently of how confident or detailed the trace reads, because the faithfulness research shows a trace can be structurally sound and causally disconnected from the actual answer at the same time [S3]. Amateurs assume "think step by step" prompting is a universal trick; pros know its payoff is uneven — strongest in larger models on certain task types [S1] — and it tends to matter less on models already tuned with dedicated reasoning modes, where the equivalent behavior is closer to built in. Above all, pros never confuse "the model generated a lot of reasoning tokens" with "the model reasoned correctly" — the first is a token count, the second is a claim about truth, and closing that gap takes independent verification, not a longer trace.
