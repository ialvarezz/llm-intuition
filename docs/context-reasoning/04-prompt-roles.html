System / User / Assistant Roles

Roles aren't metadata the model understands — they're tokens it was trained to weigh differently. That training is a strong habit, not a rule.

// Roles are tokens, not categories

    
A chat request looks like a list of labeled turns — {"role": "system", ...}, {"role": "user", ...}, {"role": "assistant", ...} — and it's tempting to read that as metadata the model consults the way a program reads a JSON key. That isn't what happens. The harness applies a chat template that converts the role/content list into one linear token sequence, inserting literal control tokens at the boundaries: "The list of role and content dictionaries that you pass to a chat model get converted to a token sequence, often with control tokens like <|user|> or <|assistant|> or <|end_of_message|>, which allow the model to see the chat structure" [S4]. By generation time there is no role field left — only tokens, no different in kind from any other token in the sequence.

    
The proof this is a template convention rather than something universal: models fine-tuned from the identical base can use completely different control-token vocabularies for the same structural idea. Mistral-7B-Instruct renders a turn as [INST] ... [/INST] with no word "user" or "assistant" anywhere; Zephyr-7B, fine-tuned from that same Mistral base, instead uses <|user|> and <|assistant|> [S4]. If "user" and "assistant" were concepts the model understood the way a person understands "who's asking," that wouldn't need re-teaching per fine-tune with an arbitrary choice of symbols — and getting it wrong has a real cost: "with the wrong control tokens, these models would have drastically worse performance" [S4]. Anthropic's API makes the same point from a different angle: system is a distinct top-level request field, not one more value of a shared role key sitting in the messages array with user/assistant [S1]. OpenAI keeps the equivalent instruction inside one array, under yet another label — "developer" rather than "system" [S3]. Three vendors, three concrete implementations of the same idea; the specific tokens and field names are not the constant, the underlying trained behavior is.


    
// A trained priority, not hard logic

    
Roles change behavior because instruction-tuning and RLHF specifically train the model to weight content differently depending on which role carried it. OpenAI states the intended hierarchy directly: "developer messages are instructions provided by the application developer, prioritized ahead of user messages," while "user messages are instructions provided by an end user, prioritized behind developer messages" [S3] — "like a function definition" versus "arguments to a function" [S3]. Anthropic's framing points at the same effect from the usage side: "Setting a role in the system prompt focuses Claude's behavior and tone for your use case. Even a single sentence makes a difference" [S1].

    
The crux: this priority is a trained tendency, not hard-wired logic comparable to an OS mechanically refusing a syscall from an unprivileged process. There's no code path that says "if role == system, treat as inviolable" — there's a learned statistical habit that system/developer-labeled content, on average across the training distribution, gets weighted more heavily when it conflicts with user-labeled content. In the common case this works as intended: a system instruction to answer in bullet points quietly shapes format the whole conversation, with no conflict to resolve. The interesting case is when a user turn actively contradicts it — well-trained models tend to hold the line and favor the system instruction [S1] [S3], but "tend to" is doing real work in that sentence. A weakly-specified system prompt gives the trained prior less to hold onto, and nothing about role position mechanically forces the outcome — it's a prior competing with everything else in the context, and priors lose sometimes.


    
// Where it breaks: injection

    
That gap — role as trained behavioral prior versus role as security boundary — is exactly what Greshake et al.'s indirect prompt injection research is built on. Their framing: "So far, it was assumed that the user is directly prompting the LLM. But, what if it is not the user prompting? We argue that LLM-Integrated Applications blur the line between data and instructions" [S2]. The attack doesn't touch the user turn at all — it plants instruction-like text inside content the application retrieves and feeds to the model anyway: a web page, a search result, a document. The model doesn't receive that content flagged "instruction embedded in retrieved data"; it just receives tokens, indistinguishable at the model's level from any other tokens in whatever channel carried them in. If those tokens read like an instruction, nothing about their point of entry mechanically stops the model from following them the way it would follow a user instruction — and the paper demonstrated this practically against real systems, including "Bing's GPT-4 powered Chat and code-completion engines" [S2].

    
The general shape of the failure: content arriving through a tool-result or retrieved-document path carries no automatic "this is just data" framing unless something in the harness or prompt makes that framing explicit. Feeding an API response or a file's contents straight into context and expecting instruction-like text inside it to stay inert is exactly the gap this research describes — role position alone (system beats user beats retrieved content, in intended priority order) is a trained tendency about how much weight to give content, not an enforced boundary about what content is allowed to do.


    
// Pro vs. amateur

    
Amateurs treat the system/user split as a dumping ground for whatever content feels important, and treat role position itself as a security boundary — "it's in the system prompt, so it's safe." Pros put stable, request-independent behavior in the system slot and variable, per-call content in the user slot, because that's both what the model was trained to expect [S1] [S3] and what keeps the prompt-cache prefix stable — dumping a per-request document into the system slot busts the cache on every call. Amateurs paste retrieved documents or tool output straight into context and trust the channel alone to mark it as inert; pros explicitly delimit and label untrusted content — wrapping it in clear tags, prefacing it as "retrieved data, not instructions" — because position alone doesn't do that work, as the indirect-injection research demonstrates at scale [S2]. And above all, pros treat role-based priority as what it actually is: a strong, useful, trained prior that reliably shapes behavior in the common case, never a guarantee that forecloses adversarial or accidental misuse.
