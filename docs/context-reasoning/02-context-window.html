Context & the Context Window

The window isn't just a size limit — what's in it, in what order, shapes every answer long before it's ever full.

// Everything counts, every turn

Module 00 introduced the context window as a size limit. Anthropic's own definition is precise and worth restating: the context window "refers to all the text a language model can reference when generating a response, including the response itself," representing "a 'working memory' for the model," distinct from the training corpus [S1]. Windows top out "up to 1M tokens, depending on the model," and that ceiling holds "the conversation history plus the new output" the model generates [S1] — the output isn't extra room, it's carved from the same budget as the input.

What actually counts toward that budget is broader than the visible chat log: "Everything in the request counts toward the context window: the system prompt, every message in messages (including tool results, images, and documents), and your tool definitions. The output Claude generates for the turn, including its extended thinking, counts too" [S1]. Tool schemas are sent every turn whether or not a tool gets used. As a conversation advances, "each user message and assistant response accumulates within the context window, and previous turns are preserved completely" in the raw API [S1] — chat interfaces like claude.ai instead manage the window "on a rolling 'first in, first out' basis" [S1], quietly dropping old turns as a UX choice the underlying model doesn't make on its own. And caching doesn't shrink the budget: "cached prompt prefixes still occupy the context window" [S1] — caching changes what you pay, not what counts.

// Composition before capacity

Order. Liu et al.'s "Lost in the Middle" study found that language-model performance "is often highest when relevant information occurs at the beginning or end of the input context, and significantly degrades when models must access relevant information in the middle of long contexts, even for explicitly long-context models" [S2]. This is a positional-recall problem, not a capacity one — the tokens are all technically in the window, well under any cap. Early instructions buried under pages of later volume sit in the structurally worst position, which bridges directly into concept 03's attention curve.

Redundancy. Repeated or contradictory context doesn't just waste budget — competing copies of the same fact become competing candidate "truths" the model has to arbitrate at generation time, with no principled signal for which is current.

Relevance mix. Anthropic frames context as "a critical but finite resource for AI agents" [S4], tracing the cost to the architecture itself: transformers let "every token attend to every other token across the entire context," producing "n² pairwise relationships for n tokens" [S4]. More tokens — even irrelevant, even with room to spare — means more competing relationships. Anthropic calls the resulting degradation "context rot": "as the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases," a pattern that "emerges across all models," though "some models exhibit more gentle degradation than others" [S4].

// Four ways to handle overflow

1. Truncation / eviction. Drop the oldest content to make room — the same FIFO behavior Anthropic describes for chat interfaces [S1]. Cheap, no extra computation, completely lossy: nothing in the remaining context records what was dropped.

2. Sliding window. Structurally the same loss as truncation, but deliberate: keep only the last N turns by policy. LangChain frames the underlying problem plainly — "a full history may not fit inside an LLM's context window, resulting in an context loss or errors," and even generous windows don't fix it, since "most LLMs still perform poorly over long contexts" [S6]. Their trimming approach counts tokens and keeps "only the last few messages to fit context window" [S6] — bounded cost, total loss past the edge, and LangChain says so directly: "the problem with trimming or removing messages... is that you may lose information from culling of the message queue" [S6].

3. Summarization / compaction. Compress old turns into a running summary instead of dropping them. Anthropic calls this "compaction": "the practice of taking a conversation nearing the context window limit, summarizing its contents, and reinitiating a new context window with the summary" [S4], often "the first lever in context engineering to drive better long-term coherence" [S4]. It can be selective — Anthropic's own agent "preserves architectural decisions, unresolved bugs, and implementation details while discarding redundant tool outputs or messages" [S4] — but "the art of compaction lies in the selection of what to keep versus what to discard, as overly aggressive compaction can result in the loss of subtle but critical context" [S4]. LangChain's vendor-neutral framing lands on the same trade-off: "some applications benefit from a more sophisticated approach of summarizing the message history using a chat model" [S6] precisely because trimming alone loses too much.

4. RAG / offload. Never front-load the material at all. Agents built "just in time" instead "maintain lightweight identifiers (file paths, stored queries, web links, etc.) and use these references to dynamically load data into context at runtime using tools" [S4]. The window holds pointers, not payloads — sidestepping capacity and composition problems at the cost of retrieval latency and the risk of missing something relevant.

// Pro vs. amateur

Amateurs treat the context window as a bucket to fill — if it fits, it can only help. Pros treat it as a budget to curate: "more context isn't automatically better," because "as token count grows, accuracy and recall degrade," so "curating what's in context [is] just as important as how much space is available" [S1]. Amateurs let the framework's default decide what happens on overflow; pros pick a strategy deliberately, matched to the task — a sliding window for a short chat, compaction for a long agentic session, RAG offload for a task sitting on a large mostly-irrelevant corpus. Amateurs assume content that fits under the cap is free; pros know redundant context isn't free even with room to spare — it competes for the same n² attention budget as everything else [S4], and three copies of a fact aren't reinforcement, they're three candidate truths to arbitrate. Context stuffing is the expensive alternative to retrieval, not the cheap one — it's priced per token whether or not it's used [S4].
