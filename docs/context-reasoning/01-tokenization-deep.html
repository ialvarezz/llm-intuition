Tokenization, Deeper

Module 00 showed that a model never sees words. Here's the mechanism that decides what it sees instead — and why chat roles, cost, and weird failures all trace back to it.

// BPE, step by step

The dominant scheme behind GPT-family, Llama, and Mistral tokenizers is byte pair encoding (BPE), and it's worth being precise about what "learned vocabulary" means. Start from the smallest possible alphabet: raw bytes. Modern BPE tokenizers work byte-level, so before any merging happens, an input string is just its UTF-8 byte sequence — 256 possible symbols, IDs 0 through 255 [S1]. Then the algorithm scans a large training corpus, counts how often every adjacent pair of symbols occurs, and merges the single most frequent pair into a new symbol, added to the vocabulary as the next token ID. The corpus is rescanned with that merge applied, frequencies are recounted, and the process repeats — one merge per step — until the vocabulary hits a target size, typically tens of thousands to low hundreds of thousands of entries [S1][S3][S4]. tiktoken's production encodings are a concrete instance: cl100k_base and o200k_base are named for roughly their vocabulary sizes, and different OpenAI model families use different named encodings — there is no single, universal "the" GPT tokenizer [S4].

The word to sit with is learned. This is a statistical compression procedure run once over a training corpus, then frozen and shipped with the model — not a linguist's rule set. tiktoken's own README gives a clean illustration: because "ing" is a common English subword, BPE will often split "encoding" into tokens like "encod" and "ing" [S4] — the split falls where statistics found a boundary, not where a grammarian would put one. BPE itself predates LLMs: Sennrich et al. (2015) introduced it for machine translation, framing the core problem as translation being open-vocabulary while models want a fixed vocabulary [S3] — the same trade-off that makes it the right tool for tokenizing arbitrary internet text today.

// Vocabulary size is a trade-off

A bigger vocabulary absorbs more of a given text into single tokens rather than several — tiktoken's BPE compresses so each token corresponds to roughly 4 bytes of original text on average [S4]. Fewer tokens per input means shorter sequences: cheaper to process, and since pricing and context limits are denominated in tokens, cheaper to pay for. But every vocabulary entry needs a row in the embedding matrix and a row in the output matrix — more vocabulary is more parameters and more compute at exactly the two layers every forward pass touches. Shrink the vocabulary instead and those matrices shrink, but sequences get longer for the same text, and transformer attention cost scales with sequence length too. No free lunch — which is exactly why different model families ship different vocabulary sizes and different named tiktoken encodings for different model generations [S4], rather than converging on one number.

// Special tokens are real tokens

When you chat with a model through a UI, you see a clean back-and-forth. What the model consumes is one flat token sequence, and the role structure is represented inside it by literal vocabulary entries — control tokens like <|user|>, <|assistant|>, or bracket markers like [INST]/[/INST] — inserted around each message's content before tokenization [S2]. Hugging Face's chat template docs make this concrete: Mistral-7B-Instruct and Zephyr-7B are fine-tuned from the same base model, but use entirely different control-token schemes for the same role structure [S2]. There's no universal "role" baked into the architecture — it's populated by whichever tokens that model's chat template was fine-tuned to expect, and getting them wrong doesn't degrade gracefully: it can mean the model continues the user's message instead of replying to it [S2]. These tokens are real vocabulary entries too — every role marker and generation-prompt token consumes a slot in the sequence and counts against the context window and the token-metered price, invisible in the chat bubble but not invisible in the bill.

// Pro vs. amateur

Amateurs think in words and characters; pros think in tokens, because that's the actual unit cost, context budget, and model behavior are all measured in. Amateurs assume tokenization is a fixed, universal fact about a string; pros know the same text produces different token counts on different tokenizers — different model families ship different named encodings for exactly this reason [S4], and recounting against the model you're actually using rather than reusing a stale count is standard practice, not paranoia. Amateurs think a chat prompt is "the messages"; pros know it's the messages plus the chat template's control tokens — real, invisible-in-the-UI cost that's billed and budgeted exactly like visible text [S2]. And amateurs are baffled when a model mangles a long number or a rare username, or can't reliably spell a word; pros recognize a rare string fragmenting into unfamiliar chunks, and remember Karpathy's blunt framing — "Why can't LLM spell words? Tokenization." [S1] — the model never had character-level access to begin with.
