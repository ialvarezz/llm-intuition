1 · Tokenization

Two famous failures, one cause

"How many r's in strawberry?"
The model never sees ten letters. It sees roughly strawberry — three opaque IDs. Nothing in an ID encodes "contains two r's," so counting is a memory recall about spelling, not perception. Ask it to write s-t-r-a-w-b-e-r-r-y first and each letter becomes its own visible token — suddenly it can count.
Why is arithmetic shaky?
Token boundaries ignore place value: 1234 + 567 — the 4 and the 7 are both "ones" digits but sit in unrelated chunks, so columns can't align. Writing the digits out one at a time rebuilds the alignment in context — which is exactly what step-by-step reasoning does.
Same root cause: the token is the smallest unit the model can perceive. Everything inside one is invisible unless it was separately memorized. Both fixes work by re-spelling the problem into smaller tokens — remember this slide when we get to reasoning.

  3 · Attention curve
  
In the window isn't the same as seen

  
  

    
    
    
    
    
  

  

  
Why the U-shape: attention weights are normalized across every token in the window — more tokens competing means less weight available per token, so signal spreads thin. Training adds the bias: sequence starts dominate training data (primacy) and the causal mask keeps recent tokens close (recency). The middle gets neither. Measured by Liu et al. 2023 across models built specifically for long context — the effective window is smaller than the advertised one.

  





  3 · Attention curve
  
Why position mechanically matters

  

  

  
  
Each bar is the highlighted token's attention share paid to one earlier token — the percentages sum to 100% across the row, because attention is a fixed budget being divided, not a set of independent scores. A token that gets a thin slice is functionally invisible on this step even though it sits in the window. Watch "it": most of its budget goes to "ball" — that's coreference happening — but stretch the sentence to thousands of tokens and that slice thins toward nothing. This per-token budget-splitting is the mechanism behind the U-curve on the previous slide.

  4 · Roles
  
Who actually writes into each role?

  

    
System — the operator's channel
      
Written by the application or harness developer, not the person chatting. Populated two ways: static configuration, or content the harness loads on demand — a Skill that injects the team's instructions when relevant is literally writing into the system role. (How Skills work: Module A1.)

    
User — the human's channel, and the danger zone
      
The person's actual ask, every turn — but tool results, retrieved pages, and file contents usually land here too. That puts untrusted text in the seat the model treats as a person speaking, which is the entire mechanism of prompt injection. Delimit it and label it as data.

    
Assistant — the model imitates itself
      
The model's own prior turns. It doesn't just remember them — it conditions on them as examples: consistency and persona drift are the model pattern-matching its earlier style. That's also why prefill works: put a partial answer in the assistant role and the model continues it — an opening { effectively forces JSON. (Some current APIs now reject prefill on their newest models — but the imitation mechanism is why it ever worked.)

  

  
Roles are a privilege system: operator constraints → system, task content → user, and anything from outside the trust boundary gets marked as data no matter which envelope carried it in.






  4 · Roles
  
Pick the role by the job

  

    
I want behavior that holds for the whole session…
      
System. Stable instructions, style rules, persona — set once, applies every turn.

    
I want to ask something specific right now…
      
User. The actual task or question — changes every turn.

    
I want to show an example of the response style I want…
      
Assistant (few-shot). A prior turn the model treats as its own past output — shapes tone by example.

    
I'm passing in untrusted content (a doc, a tool result)…
      
User or tool role, explicitly delimited. Never let it read as system or assistant — that's how indirect prompt injection works.

  






  5 · Reasoning
  
More tokens, not a different process

  

    
    
  

  

  
Where does the trace come from? A transformer spends a fixed amount of compute per generated token — one forward pass, same depth, whether the question is trivial or hard. Generating intermediate tokens is therefore the only way the model can spend more compute on a harder problem: each step is another pass, and every step it writes becomes readable input for the next. The trace is ordinary generated text used as external scratch memory — the same trick as spelling out s-t-r-a-w-b-e-r-r-y, rebuilding structure the tokenizer destroyed. That's also exactly why it costs more tokens.

  





  5 · Reasoning
  
Effort is a dial, not a cap

  
  

  
Hard token caps are the old model — current APIs use adaptive thinking: the model decides when and how much to think, and effort shifts that threshold. The catch: the decision to think is itself a next-token prediction, keyed on surface features that correlated with difficulty in training. "How many r's in strawberry" is seven words and looks trivial — nothing on its surface signals that it fights the tokenizer — so it gets a fast, confident, wrong answer. Don't let the model's self-assessment be the router on paths that matter.

  





  5 · Reasoning
  
A trace isn't proof

  

    
Confident wrong reasoning
      
Fluent, structured, step-numbered — and the arithmetic in step 2 is wrong. Structure reads as credibility; it isn't.

    
Trace doesn't match the final answer
      
The steps derive one value, the stated final answer is a different one — the trace and the output can diverge.

    
Reasoning as elaborate hallucination
      
More steps can mean more confident fabrication when the model lacks the underlying fact, not less.

  






  Putting it together
  
One conversation, all five mechanisms

  

    
1 · Tokenize: the prompt and every prior turn become tokens the moment they're sent — nothing is "words" past this point.

    
2 · Window fills: system prompt, history, a pasted doc, and reserved output all draw from the same budget.

    
3 · Attention curve bites: a fact buried mid-document gets less effective weight than one at the start or end — even though it's technically in-window.

    
4 · Roles structure the ask: the system prompt sets the stable behavior, the user turn carries the actual question, and the harness assembles both into one sequence.

    
5 · Reasoning produces the answer: the model spends extra tokens working through the multi-step part before committing to a final response.

  






  BKMs
  
What to actually do

  

    
Tokenization
      
Budget in tokens, not words. Expect code and non-English text to cost more per character.

    
Context window
      
Curate what's in the window — pick an overflow strategy on purpose, don't just let the default truncate.

    
Attention curve
      
Don't bury the critical instruction in the middle of a long prompt. Restate it near the end.

    
Roles
      
Stable behavior → system. The actual ask → user. Never let untrusted content read as either.

    
Reasoning
      
More reasoning tokens is a cost/quality knob — verify conclusions, don't trust a confident trace.
