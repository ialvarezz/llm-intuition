# 03 · Embeddings

## From symbols to geometry

Note 02 stopped at integers: a tokenizer turns your text into a sequence of token IDs, each one an index into a fixed vocabulary. An integer ID by itself carries no meaning a neural network can compute with — token 4187 isn't *closer* to token 9021 in any useful sense just because 4187 < 9021. Before a transformer can do anything useful with a token, that ID gets mapped to an **embedding**: a learned list of floating-point numbers — a vector — that positions the token as a point in a high-dimensional space. Google's ML crash course frames the underlying problem directly: a one-hot vector (one slot per vocabulary entry, all zeros except a single 1) means "large input vectors mean a huge number of weights for a neural network," and worse, "one-hot encoding vectors lack meaningful relationships, failing to capture semantic similarities between items" [S3]. An embedding fixes both problems at once — it's built as "dense vector representations that capture semantic relationships and reduce the dimensionality of data" instead of throwing that structure away [S3].

Crucially, the numbers in an embedding aren't hand-designed. Nobody decided that dimension 47 means *royalty* and dimension 812 means *edibility*. The vector for each token is a set of weights the model learns during training, the same way it learns everything else — by adjusting numbers to reduce prediction error over enormous amounts of text. The 2013 word2vec paper that popularized this whole approach describes the goal plainly: "novel model architectures for computing continuous vector representations of words", judged by how well those vectors do on "measuring syntactic and semantic word similarities" [S1]. Words (or, in a modern LLM, subword tokens) that show up in similar contexts end up with similar vectors — not because a human labeled them similar, but because the training process pulled them together.

## Distance is meaning

Once tokens are points in space, *similar meaning* becomes *nearby points*. This is the whole trick: it converts a fuzzy linguistic question (do these two things mean roughly the same thing?) into an arithmetic one (how far apart are these two vectors?). OpenAI's embeddings guide states this as directly as it can be stated: for two embedding vectors, "small distances suggest high relatedness and large distances suggest low relatedness" [S2].

The specific way distance gets measured is usually **cosine similarity** — instead of measuring the straight-line gap between two vectors, it measures the angle between them, ignoring how long each vector is and asking only whether they point in the same general direction. Two vectors pointing almost the same way score close to 1 (very similar); two pointing in unrelated directions score close to 0; two pointing in opposite directions score close to -1. You don't need the formula to use the intuition: it's a compass-bearing comparison, not a ruler comparison. OpenAI's guidance is simply "We recommend cosine similarity" [S2] — its embeddings are normalized to length 1, which means "Cosine similarity can be computed slightly faster using just a dot product" and "Cosine similarity and Euclidean distance will result in the identical rankings" [S2]: for vectors scaled to the same length, cosine similarity and straight-line distance rank things in the same order anyway, so cosine similarity is the cheaper way to get the same answer. Anthropic's embeddings documentation makes the same point about its recommended third-party provider, Voyage AI: "Voyage AI embeddings are normalized to length 1, which means that: Cosine similarity is equivalent to dot-product similarity... Cosine similarity and Euclidean distance will result in the identical rankings" [S4].

The most famous demonstration that this geometry captures more than *same topic* is vector arithmetic on word embeddings: take the vector for *king*, subtract the vector for *man*, add the vector for *woman*, and the closest point in the space to the result is *queen* — commonly written `king - man + woman = queen`. This specific example comes from the word2vec research line associated with the 2013 paper family [S1][S5] — the underlying claim is that a relationship like *male royal → female royal* gets encoded as a consistent direction in the vector space, one that transfers across different starting words. It's a genuinely striking result, and it's also easy to over-trust. The same mechanism that produces that equation also produces whatever biases sit in the training data — a well-known write-up on this exact analogy warns that "all results are a function of the data we used to feed our algorithm, not objective truth; so it is easy to get stereotypes like `doctor - man + woman = nurse`" [S5]. The same source is blunt about how far to trust the geometric metaphor at all: "linear space of meaning is a disputed concept" [S5]. Treat the arithmetic as a vivid illustration of what training pulls into the geometry — including its prejudices — not as proof that meaning is literally linear algebra.

## Token vs. document embeddings

There are two things people call *embeddings*, and mixing them up is a common source of confusion.

**Token embeddings** live inside the model. Every token in the vocabulary has a learned vector, stored in an embedding table that's part of the model's weights — this is the very first step a transformer performs on your input, before attention (note 04) or anything else runs. These vectors get further transformed layer by layer as the model processes a sequence; you never call an API to get one, and you never see the raw numbers.

**Sentence or document embeddings** are what you get from a dedicated **embedding model**, called through an API or run locally, that takes an arbitrary chunk of text — a sentence, a paragraph, a whole document — and returns a single fixed-length vector representing the whole thing. This is what OpenAI's guide describes: an embedding is "a vector (list) of floating point numbers" produced for a string of your choosing, useful for "Search (where results are ranked by relevance to a query string)" among other tasks like clustering and anomaly detection [S2]. Anthropic's own docs open with the same framing at the text level: "Text embeddings are numerical representations of text that enable measuring semantic similarity" [S4].

Same underlying idea — proximity in a learned vector space encodes similarity — applied at two different granularities: one number per token, deep inside a language model's forward pass; one number per chunk of text, from a separate, purpose-built model you call directly.

## Why this matters for the roadmap

Sentence/document embeddings are the entire foundation of **semantic search** and **retrieval-augmented generation (RAG)**, which Module 2 covers in depth. The pattern is always the same: embed a large collection of documents once, ahead of time; embed a user's query the same way, at request time; then find whichever stored documents have embeddings closest to the query's embedding, and use those as the most relevant results. Anthropic's own quickstart example for embeddings does exactly this — it embeds a small corpus of documents, embeds a query, then runs "a nearest neighbor search to find the most relevant document based on the distance in the embedding space" [S4], and the closest document is the one that actually answers the query. RAG builds directly on top of that: retrieve the nearest documents to a question, then hand them to a chat model as context so it can answer using material it wasn't necessarily trained on. None of it works without the core claim from this note — that distance in embedding space is a usable proxy for two texts being about the same thing.

## Pro vs. amateur

**Amateurs assume an LLM only has one kind of intelligence at work. Pros know embedding models and chat models are different products.** Anthropic — a company that ships some of the most capable chat models available — ships zero embedding models of its own — "Anthropic does not offer its own embedding model" — and instead directs developers to a separate specialist vendor, Voyage AI, telling them "you should assess a variety of embeddings vendors to find the best fit for your specific use case" [S4]. Picking an embedding model is a separate decision from picking a chat model, made on separate criteria — Anthropic's own guidance lists dataset size and domain specificity, inference performance, and customization as what actually matters [S4]. This split also shows up in cost: OpenAI's smaller embedding model processes roughly 62,500 pages of text per dollar, versus 9,615 pages per dollar for its larger sibling [S2] — embeddings are, model-for-model, priced for bulk use in a way chat completions rarely are, because embedding a document once is a much smaller computation than generating a paragraph of new text.

**Amateurs mix and match vectors from different sources in one index. Pros never compare embeddings across models.** A distance calculation between two vectors is only meaningful if both vectors were produced by the same model — with the same learned weights and, often, the same output dimension. Voyage AI's models, for example, come in variants with different dimension counts (256, 512, 1024, 2048) depending on which model and configuration you pick [S4]; a 1024-dimension vector and a 3072-dimension vector aren't even comparable as numbers, let alone comparable as meanings. Building a search index with embeddings from one model, then querying it with embeddings from a different model (or even a different snapshot of the same model), silently produces meaningless nearest-neighbor results — no error, no warning, just wrong answers that look plausible.

**Amateurs treat semantic similarity as an objective property of language. Pros know it's a trained property of a specific model.** Embedding models come in domain-tuned flavors for exactly this reason — Anthropic's recommended provider ships `voyage-code-3`, "Optimized for **code** retrieval", alongside separate models tuned for legal and finance text [S4]. A general-purpose embedding model and a code-tuned one will rank the same pair of snippets differently, because what counts as similar was learned from different training data. There is no view-from-nowhere embedding space; there's only similar according to this model, trained on this data.

**Amateurs treat higher dimensionality as strictly better. Pros know it's a cost/nuance trade-off, tuned per use case.** Embedding dimension is an adjustable knob, not a fixed property — OpenAI's `text-embedding-3-large` defaults to 3072 dimensions versus 1536 for the smaller model, and both expose a `dimensions` parameter to shrink the vector further "without sacrificing concept representation" [S2]; Voyage's models similarly offer the same underlying model at 256, 512, 1024, or 2048 dimensions [S4]. More dimensions can capture finer distinctions between texts, but every dimension costs storage, memory, and search latency at scale — a production system indexing millions of documents will deliberately trade some nuance for a smaller, faster vector rather than defaulting to the largest option available.

**Amateurs read `king - man + woman = queen` as proof embeddings understand meaning. Pros read it as a demonstration of what training data encodes, biases included.** The analogy is real and reproducible, but the same geometric trick that finds *queen* from *king*, *man*, *woman* is equally capable of encoding whatever stereotypes exist in the source text, producing results like `doctor - man + woman = nurse` [S5]. The person who wrote up this exact example is explicit that the whole framing rests on a debatable premise: "linear space of meaning is a disputed concept" [S5]. Pros treat vector arithmetic as a useful intuition for how embedding spaces organize relationships, not as evidence that a model literally understands gender roles or royalty in any human sense.
