# 01 · Plugin marketplaces

## Why extend an assistant at all

Concept 00 established the frame: a plugin is not a change to the model, it's a change to the harness — a new tool exposed, a new piece of context injected, a new command recognized. The question this note answers is where those additions come from and how they get onto your machine.

The lazy path is repeated prompting: pasting the same instructions, the same project context, the same "review this like a security engineer" preamble into every session. That works, but it doesn't survive a teammate joining, a new machine, or six months passing — it's copy-paste engineering, and copy-paste engineering doesn't version, doesn't share, and doesn't get reviewed. Packaging that same set of instructions, tools, and hooks into a plugin turns it into an artifact: something with a name, a version, a source you can point at, and a diff someone else can read before they run it. Claude Code's plugin docs describe exactly this shift — plugins are "a lightweight way to package and share any combination of slash commands, subagents, MCP servers, and hooks for Claude Code customization" [S3]. Once something is packaged, it can be distributed. Distribution is what a marketplace is for.

## A catalog, not a store

The word "marketplace" invites the wrong mental model — a hosted storefront, like an app store, with the platform holding the inventory. That's not what it is here. Claude Code's own definition is precise: a "plugin marketplace is a catalog that lets you distribute plugins to others. Marketplaces provide centralized discovery, version tracking, automatic updates, and support for multiple source types, including git repositories and local paths" [S1]. A marketplace is metadata — a list of pointers — not a repository of code it hosts itself.

Concretely, a Claude Code marketplace is a git repository (or a URL, or a local path) containing a file at `.claude-plugin/marketplace.json`. That file has three required top-level fields: `name` (a kebab-case identifier — "This is public-facing: users see it when installing plugins" [S1]), `owner` (an object with a required `name` and optional `email` for the maintainer), and `plugins` (the array of listed plugins) [S1]. Marketplace requirements are stated the same way in the launch blog post: "a git repository, GitHub repository, or URL with a properly formatted `.claude-plugin/marketplace.json` file" [S3].

Each entry in that `plugins` array requires only `name` (kebab-case plugin identifier) and `source` (a string or object describing where to fetch the plugin from) [S1]. `source` is where the "not hosting" property becomes visible: it supports a relative path (a local directory inside the marketplace repo itself, e.g. `"./my-plugin"`, which "Must start with `./`"), a `github` source (`repo`, optional `ref`, optional `sha`), a `url` source (git URL, same optional `ref`/`sha`), a `git-subdir` source (a subdirectory within a git repo, via `url` + `path`), or an `npm` source (`package`, optional `version`, optional `registry`) [S1]. A single marketplace file can list plugins that live in five different repositories owned by five different people — the marketplace only records where to look.

Optional per-plugin fields round out what a catalog entry can carry: `displayName` (falls back to `name`), `description` ("Brief plugin description"), `version` (skip it and updates fall back to the git commit SHA), `author`, `homepage`, `repository`, `license` (SPDX identifier), `category`, `tags`, and `strict` (defaults to `true`) [S1]. Version resolution, when multiple sources could supply one, follows a fixed order: the plugin's own `plugin.json` `version` field first, then the marketplace entry's `version`, then the git commit SHA of the source [S1]. The task brief's figure simplifies this down to the three fields users actually read before installing — `name · source · description` — which matches how the dump frames the required-plus-most-visible fields.

## Adding one, installing one

Getting a plugin onto your machine is a two-step handshake, matching the marketplace/plugin distinction above. First, point Claude Code at a marketplace: `/plugin marketplace add <source>` [S1], or in the shorthand the launch post uses for a GitHub-hosted marketplace, `/plugin marketplace add user-or-org/repo-name` [S3]. That step does nothing but register the catalog — it doesn't install anything yet. Second, install an individual plugin out of that catalog by name: `/plugin install <plugin-name>@<marketplace-name>` [S1]. The `@marketplace` suffix disambiguates, since the same plugin name could in principle appear in more than one catalog you've added.

A few adjacent commands round out the lifecycle: `/plugin marketplace update` refreshes a registered catalog, and the same operations are available outside the interactive session via the CLI — `claude plugin marketplace list [--json]` and `claude plugin marketplace add <source> [--scope <scope>] [--sparse <paths...>]` [S1]. Once installed, plugins aren't copied in place and forgotten — "When users install a plugin, Claude Code copies the plugin directory to a cache location at `~/.claude/plugins/cache`" [S1], and installed plugins can be toggled: the launch post describes users being able to "toggle on and off as needed" to manage system-prompt complexity [S3]. That toggle matters for the same reason Concept 00's context-window discussion mattered — every installed-and-enabled plugin's tool descriptions sit in the context window on every turn, whether or not the model ends up calling them.

Two official Claude Code marketplaces exist as a starting point rather than the only option: `claude-plugins-official`, "A curated set of plugins maintained by Anthropic" and `claude-community`, "The public community marketplace where third-party submissions land after review" [S2]. Submitting a plugin for community review goes through a form on claude.ai (for Team/Enterprise orgs) or the Console, and `claude plugin validate` checks a plugin locally before submission [S2]. Marketplace names in that reserved list — `claude-code-marketplace`, `claude-code-plugins`, `anthropic-marketplace`, and a dozen others — can't be claimed by a third-party catalog, which keeps the official/community distinction from being spoofed by name alone [S1].

## The Copilot / VS Code side

The other two ecosystems in this module's research solve overlapping problems with different shapes. VS Code has had an extension marketplace since long before Claude Code plugins existed, and its model is closer to what "marketplace" usually connotes: "VS Code extensions let you add languages, debuggers, and tools to your installation to support your development workflow" [S4], installed to a fixed location per platform (`~/.vscode/extensions` on macOS/Linux, `%USERPROFILE%\.vscode\extensions` on Windows) [S4] and identified by a `publisher.extension` string, e.g. `wayou.vscode-todo-highlight` [S4]. Command-line management mirrors the Claude Code CLI shape — `code --install-extension <publisher.extension>`, `code --list-extensions`, `code --uninstall-extension <extension-id>` [S4] — and, notably, VS Code added an explicit trust gate in version 1.97: "when you first install an extension from a third-party publisher, VS Code shows a dialog prompting you to confirm that you trust the extension publisher" [S4]. A workspace can also declare recommended extensions for anyone who opens it, via `.vscode/extensions.json`'s `recommendations` array [S4] — the closest VS Code analog to a team pointing new members at a private marketplace.

GitHub Copilot's extensibility story doesn't route through a marketplace-of-extensions model at all — it routes through MCP. The docs describe MCP as "an open standard that defines how applications share context with large language models (LLMs)," which enables users to "extend the capabilities of GitHub Copilot by integrating it with other systems," and the same protocol "works across all major Copilot surfaces—whether you're working in an IDE, using GitHub Copilot CLI, working in the GitHub Copilot app, or delegating tasks to an agent on GitHub.com" [S5]. So where Claude Code has a `marketplace.json` catalog of plugins and VS Code has a publisher-namespaced extension gallery, Copilot's "where extensibility actually lives" is the set of MCP servers you connect — local ones supported in VS Code, JetBrains IDEs, and Xcode, remote ones (over OAuth or a PAT) supported more broadly, including Cursor and Windsurf [S5]. The GitHub-hosted MCP server and a Playwright MCP server are configured by default on GitHub.com [S5], and the MCP policy that governs which servers an org can connect applies only to Copilot Business/Enterprise, excluding Free, Pro, Pro+, and Max tiers [S5]. Later concepts in this module cover MCP servers directly (Concept 03) and the Claude/Copilot compatibility question in more depth (Concept 08); the point here is narrower — Copilot's "marketplace" question and Claude Code's are different questions, because the two harnesses picked different extension mechanisms as their primary one.

## Trust: you are running their code

None of the mechanics above change what installing a plugin actually does: it puts someone else's instructions and, in the case of hooks or MCP servers, someone else's executable code inside your session, running with whatever access your session has. A marketplace catalog entry doesn't vet that code for you — it's a pointer, and the source it points to can be updated by its owner at any time (which is exactly what `version`/`ref`/`sha` pinning exists to control) [S1]. VS Code's third-party-publisher trust dialog [S4] is a direct acknowledgment of the same risk in that ecosystem: the platform doesn't claim to have already checked the extension for you, it makes you explicitly confirm you're choosing to trust the publisher.

The practical consequence: review a plugin's source — or at minimum its diff against a version you've already reviewed — before installing, the same way you'd review a new dependency before adding it to a project. Pin to a `ref`/`sha` rather than tracking a moving branch when the plugin matters to your workflow, and prefer private or team-curated marketplaces for anything you're deploying to a whole org, since a private repository's access controls (with token-based auth for GitHub, GitLab, or Bitbucket, for auto-updates in the background [S1]) are the actual trust boundary — not the marketplace listing.

## Pro vs. amateur

**Amateurs treat a marketplace as a store that hosts and vets plugins. Pros know it's metadata pointing at git.** The plugin's actual source can live anywhere git (or npm, or a bare URL) reaches [S1] — the marketplace only supplies discovery and a name. Nothing about being listed implies review, sandboxing, or hosting by the marketplace's owner.

**Amateurs install and forget. Pros pin and review versions like any other dependency.** `source` supports `ref` and `sha` pins on `github`, `url`, and `git-subdir` sources specifically so an update doesn't silently change what's running in your session [S1]; omitting `version` means every new commit is a new version [S2].

**Amateurs assume "official" or well-known marketplaces are the whole story. Pros know private and team marketplaces are the real enterprise use case.** The reserved-name list [S1] and the `claude-plugins-official`/`claude-community` split [S2] exist for public discovery, but nothing in the mechanism restricts a marketplace to being public — a team can host `.claude-plugin/marketplace.json` in a private repo and point only their own members at it, using the same `github`/`url` source types and the same token-based auth path Claude Code already supports [S1].

**Amateurs judge a plugin by which marketplace it's listed in. Pros read the source.** A listing in `claude-community` means a submission passed review for that marketplace [S2] — it doesn't mean the plugin's hooks are safe to run with your credentials, or that its MCP server doesn't phone home. The marketplace name on a plugin is not a substitute for reading what the plugin actually does.
