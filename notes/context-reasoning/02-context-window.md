# 02 · Context & the context window

Module 00 introduced the context window as a size limit — a number of tokens, past which something has to give. That framing is correct but incomplete, and the incompleteness matters in practice. A context window isn't a bucket that's either "has room" or "is full." It's a working memory with contents, an order, and a composition, and all three of those shape the model's output long before the bucket is anywhere near the brim. This note goes past "what happens when full" to the two things that matter earlier: what's actually in there, and what to do on purpose when it does fill up.

## Recap: the window as working memory

Anthropic's own framing is precise and worth restating exactly: the context window "refers to all the text a language model can reference when generating a response, including the response itself," representing "a 'working memory' for the model," distinct from the enormous corpus it was trained on [S1]. That distinction — working memory versus training corpus — is the one people conflate most often. A model's weights encode statistical patterns absorbed from trillions of tokens of training data; none of that is "in" the context window. The context window is only what's placed in front of the model for *this* particular generation: assembled fresh, every turn, discarded (from the model's perspective) the instant the response finishes. It doesn't remember your previous conversation from yesterday because nothing carried it forward into today's window.

The other precise fact from Module 00 worth restating with its numbers: context windows top out "up to 1M tokens, depending on the model," and that ceiling holds "the conversation history plus the new output" the model generates [S1]. The output is not extra room tacked on afterward — it's carved out of the same budget as the input. A near-full window doesn't just risk running out of room to read; it risks running out of room to answer.

## Everything counts, every turn

Here is the fact this concept exists to unpack: "Everything in the request counts toward the context window: the system prompt, every message in `messages` (including tool results, images, and documents), and your tool definitions. The output Claude generates for the turn, including its extended thinking, counts too" [S1]. Read that list again slowly, because each item is a category people forget to budget for.

The system prompt counts — including the parts a framework injects that you never typed yourself. Every message counts — not just the visible chat bubbles, but every tool result stapled into the conversation, every image, every document attachment. Tool *definitions* count — the JSON schemas describing what tools exist and how to call them are sent on every single turn, whether or not the model uses any of them that turn. And the model's own output counts, including anything it "thinks" before answering, if extended thinking is in play.

None of this is optional overhead you can wave away as "the framework's problem." A tool-using agent with a dozen tool definitions, a long system prompt, and a growing transcript of tool results is spending real, countable tokens on all four categories every single turn — and as the conversation advances, "each user message and assistant response accumulates within the context window, and previous turns are preserved completely" [S1] (in the raw API; chat interfaces like claude.ai instead manage the window "on a rolling 'first in, first out' basis" [S1], quietly dropping old turns — a UX choice, not something the underlying model does on its own). One consequence worth flagging explicitly: prompt caching changes what you *pay* for repeated prefix tokens, not whether they occupy the window — "cached prompt prefixes still occupy the context window" [S1]. Caching is a cost optimization, not a context-budget one.

## Composition matters before the cap is ever hit

This is the part "what happens when full" skips, and it's arguably the more common failure mode, because it bites well before you're anywhere near the limit.

**Order.** Liu et al.'s "Lost in the Middle" study tested how well language models retrieve relevant information at different positions within a long context, and found that "performance is often highest when relevant information occurs at the beginning or end of the input context, and significantly degrades when models must access relevant information in the middle of long contexts, even for explicitly long-context models" [S2]. This is not a capacity problem — the tokens are all technically "in" the window, well under any cap — it's a positional-recall problem. Early instructions placed at the very top of a long system prompt, then followed by pages of documents and tool results, are structurally in the worst position for recall: past the strong "beginning" effect, and probably not at the very end either. This is the direct bridge into concept 03 (the attention curve): a long-context model's attention isn't uniform across position, and a curated window has to account for that instead of assuming "it's in there, so the model will use it."

**Redundancy.** A window that repeats the same instruction three times in three different phrasings, or that accumulates multiple stale versions of a fact across a long tool-calling transcript, doesn't just waste token budget — it can create competing "truths" the model has to arbitrate between at generation time. Two different tool results describing the same file's contents at two different points in time, both still sitting in the window, are two candidate answers to "what does this file contain," and the model has no principled way to know which is current unless something in the prompt tells it. Redundant context is not neutral filler; it's active noise.

**Relevance mix.** Anthropic frames context as "a critical but finite resource for AI agents" [S4], and the underlying reason context degrades with irrelevant volume traces to the architecture itself: "LLMs are based on the transformer architecture, which enables every token to attend to every other token across the entire context. This results in n² pairwise relationships for n tokens" [S4]. More tokens in the window — even irrelevant ones, even when there's plenty of room left before any cap — means more competing pairwise relationships the attention mechanism has to weigh. Anthropic names the resulting degradation "context rot": "as the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases," and this "characteristic emerges across all models," though "some models exhibit more gentle degradation than others" [S4]. Stuffing in five tool results when the task needs one doesn't just cost tokens you'll pay for — it dilutes the model's effective attention over the one result that mattered.

The upshot: a window can be under its cap, well-ordered in theory, and still perform worse than a smaller, cleaner one, because composition — order, redundancy, relevance — is doing damage the token count alone won't show you.

## What happens when full — four strategies, not just eviction

Eventually, real conversations do hit the limit, and something has to be done about it. There isn't one answer — there are (at least) four distinct strategies, with different trade-offs, and picking one deliberately is different from accepting whatever a framework defaults to.

**1. Truncation / eviction.** The simplest response: drop the oldest content to make room for the new. This is exactly the behavior Anthropic describes for chat interfaces managing the window "on a rolling 'first in, first out' basis" [S1] — oldest turns silently fall off the front. It's cheap, requires no extra computation, and is completely lossy: whatever got dropped is gone, with no record in the remaining context that it ever existed. If the dropped content mattered later, the model has no way to know it's missing.

**2. Sliding window.** Structurally similar to truncation — old turns still disappear — but deliberate rather than incidental: keep only the most recent N turns (or N tokens) by design, as a fixed policy rather than an emergency response to hitting the cap. LangChain's guidance for chat memory frames the underlying problem plainly: "Long conversations pose a challenge to today's LLMs; a full history may not fit inside an LLM's context window, resulting in an context loss or errors," and even generous context windows don't remove the problem, since "even if your model supports the full context length, most LLMs still perform poorly over long contexts" [S6]. Their trimming approach counts tokens and truncates once the count "approaches that limit," keeping "only the last few messages to fit context window" [S6]. The cost is bounded and predictable — you always know roughly how much budget history consumes — but the loss is total for anything past the window's edge: LangChain states this directly, "the problem with trimming or removing messages... is that you may lose information from culling of the message queue" [S6].

**3. Summarization / compaction.** Instead of dropping old turns outright, compress them into a running summary and keep that instead. Anthropic calls this "compaction" in an agent context: "the practice of taking a conversation nearing the context window limit, summarizing its contents, and reinitiating a new context window with the summary" [S4], describing it as typically "the first lever in context engineering to drive better long-term coherence" [S4]. Done well, it "distills the contents of a context window in a high-fidelity manner, enabling the agent to continue with minimal performance degradation" [S4], and can be selective about what it keeps — Anthropic's own agent preserves "architectural decisions, unresolved bugs, and implementation details while discarding redundant tool outputs or messages" [S4]. The gist survives; the exact wording doesn't, and the hard part is genuinely hard: "the art of compaction lies in the selection of what to keep versus what to discard, as overly aggressive compaction can result in the loss of subtle but critical context whose importance only becomes apparent later" [S4]. LangChain's vendor-neutral framing lands on the identical trade-off from the chat-memory side: "some applications benefit from a more sophisticated approach of summarizing the message history using a chat model" [S6] precisely because trimming alone is too lossy for their needs. Compaction costs an extra summarization step (compute and latency) that pure eviction doesn't, in exchange for keeping more of what mattered.

**4. RAG / offload.** The strategy that avoids the problem at its source: never load the full material into context up front at all. Instead, "agents built with the 'just in time' approach maintain lightweight identifiers (file paths, stored queries, web links, etc.) and use these references to dynamically load data into context at runtime using tools" [S4]. The window holds pointers, not payloads, and the actual content — documents, search results, prior conversation — gets retrieved only when something in the current turn specifically needs it. This sidesteps both the capacity problem and a good chunk of the composition problem (nothing irrelevant is preloaded), at the cost of retrieval latency and the risk that the retrieval step misses something relevant that would have been trivially "just there" if front-loaded.

### Trade-off table

| Strategy | What survives | What's lost | When to use it |
|---|---|---|---|
| Truncation / eviction | Most recent content | Oldest content, silently, with no trace | Rarely a deliberate choice — usually a framework default; fine when old turns are genuinely disposable |
| Sliding window | Last N turns, predictably | Everything before the window, by design | Short-lived tasks where only recent context matters (e.g. a single support ticket) |
| Summarization / compaction | Gist, key decisions, open issues | Exact wording, fine detail, anything the summary misses | Long-running, coherence-dependent tasks — agentic coding, multi-step projects [S4] |
| RAG / offload | Nothing preloaded — only what's retrieved | Whatever isn't retrieved when needed | Large reference material (docs, codebases) that's mostly irrelevant on any given turn [S4] |

## Pro vs. amateur

Amateurs treat the context window as a bucket to fill: if it fits, put it in, more information can only help. Pros treat it as a budget to curate — Anthropic's own docs put it directly, "more context isn't automatically better," because "as token count grows, accuracy and recall degrade," and so "curating what's in context [is] just as important as how much space is available" [S1]. A window that's 40% full of redundant or irrelevant material isn't "60% to spare" — it's actively worse than a tighter one, for the composition reasons above.

Amateurs let whatever framework they're using decide what happens on overflow, and often don't know which of the four strategies it's actually running. Pros pick an eviction strategy on purpose, matched to the task: a sliding window for a short support chat, compaction for a long agentic coding session where architectural decisions from turn one still matter at turn two hundred, RAG offload for a task sitting on top of a large reference corpus that's mostly irrelevant on any given turn. The four strategies aren't ranked best-to-worst — they're different bets about what's safe to lose.

Amateurs assume that if content fits under the cap, it's free. Pros know redundant context isn't free even when there's room to spare — it competes for the same n² attention budget as everything else in the window [S4], and three copies of the same fact don't reinforce each other, they create three candidate "truths" for the model to arbitrate. And amateurs default to stuffing more material into context because retrieval feels like extra engineering work; pros recognize that context is priced per token whether or not it's used, and that just-in-time retrieval — loading a reference only when a turn actually needs it — is usually the cheaper strategy in aggregate, not the more expensive one, even though it feels like more code to write [S4].
