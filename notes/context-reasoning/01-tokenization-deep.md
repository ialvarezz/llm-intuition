# 01 · Tokenization, deeper

Module 00 established the headline fact: a model never sees words. It sees tokens — subword chunks produced by a tokenizer that sits between raw text and the network's embedding table. This note goes one level deeper: how those chunks are actually decided, why the vocabulary size is a real engineering trade-off and not a free parameter, how chat roles ride on the same token stream as everything else, and why a handful of famous "dumb" model behaviors are really tokenizer behaviors wearing a model costume.

## BPE mechanics, step by step

The dominant scheme behind GPT-family, Llama, and Mistral tokenizers is byte pair encoding (BPE), and it is worth being precise about what "learned vocabulary" means, because it's easy to wave the phrase past without absorbing it.

Start from the smallest possible alphabet: raw bytes. Modern BPE tokenizers work at the byte level, so before any merging happens, an input string is just its UTF-8 byte sequence — 256 possible symbols, IDs 0 through 255 [S1]. There is no notion of "word" or even "character" yet, just bytes.

Then the algorithm looks at a large training corpus and counts how often every adjacent pair of symbols occurs. Whichever pair is most frequent gets merged into a new symbol, which is added to the vocabulary as token 256. Then the corpus is rescanned with that merge applied, pair frequencies are recounted, and the next most frequent pair is merged — becoming token 257. This repeats, one merge per step, until the vocabulary hits a target size decided in advance, typically somewhere in the tens of thousands to low hundreds of thousands [S1][S3][S4]. tiktoken's production encodings are a concrete instance of this: `cl100k_base` and `o200k_base` are named for roughly their vocabulary sizes, and different OpenAI model families use different named encodings — there is no single, universal "the" GPT tokenizer [S4].

The word to sit with is *learned*. This is not a linguist's rule set ("split on suffixes like -ing, -ed, -tion"). It is a statistical compression procedure run once, offline, over a training corpus, and the resulting merge table is then frozen and shipped with the model. Two consequences fall out of this immediately. First, the vocabulary reflects whatever text the corpus happened to contain — common English morphemes get merged early because they're frequent, but a word or symbol string that never showed up in training gets left in smaller, less "sensible" pieces. tiktoken's own README gives a clean illustration: because "ing" is a common English subword, BPE will often split "encoding" into tokens like "encod" and "ing" [S4] — the split falls where statistics found a boundary, not necessarily where a grammarian would put one. Second, because it's frequency-driven and reversible by construction (concatenating a text's tokens recovers the original bytes exactly, lossless [S4]), the resulting vocabulary can represent any input, even strings never seen during training — worst case, it falls all the way back to individual bytes [S4]. GPT-2's tokenizer adds one more wrinkle worth knowing about: a regex pre-chunking pass that segments text before BPE runs within each chunk, specifically to prevent unwanted merges — like a word fusing with trailing punctuation — from happening in the first place [S1].

The BPE algorithm itself predates its use in LLMs by years: Sennrich et al. (2015) introduced it for neural machine translation, framing the problem as translation being fundamentally open-vocabulary while NMT models want a fixed vocabulary [S3]. Their pitch was that rare and unknown words could be represented as sequences of learned subword units instead of a special "unknown word" fallback — names, compounds, and loanwords are all translatable via smaller pieces than whole words [S3]. That same idea — a fixed-size, learned vocabulary of subword pieces that can compose to represent an open-ended space of surface strings — is exactly the property that makes BPE the right tool for tokenizing arbitrary internet text for a language model, not just parallel translation corpora.

## Vocabulary size is a trade-off

Once you see the merge target size as a knob, the natural question is: why not just crank it up? A bigger vocabulary means more of a given piece of text gets absorbed into single tokens rather than split into several — tiktoken's own BPE, for instance, compresses so that each token corresponds to roughly 4 bytes of the original text on average [S4]; a smaller vocabulary pushes that ratio the other way, toward more, smaller tokens per input. Fewer tokens per input means shorter sequences, which is cheaper: fewer forward passes through the transformer's attention mechanism to process the same amount of text, and — since API pricing and context limits are denominated in tokens, not words or characters — a direct cost and context-budget win.

But that efficiency is bought with a bigger vocabulary table, and every entry in that table needs a row in the model's embedding matrix (mapping token ID → vector) and a row in the output/unembedding matrix (mapping the model's final hidden state → a probability over every possible next token). A vocabulary of 200k tokens at a given embedding dimension is a meaningfully larger set of parameters — and more compute — at exactly the two layers every single forward pass has to touch, versus a vocabulary of 50k. Push the vocabulary size down instead, and you get the opposite trade: smaller, cheaper embedding/output matrices, but longer token sequences for the same text, since more of it gets broken into small pieces. Sequence length has its own cost, since transformer attention scales roughly quadratically with sequence length — longer sequences aren't just "more steps," they're more expensive per step too.

There is no dominant choice here, only a trade-off surface, which is exactly why different model families ship different vocabulary sizes and different named tiktoken encodings for different generations of models [S4] rather than converging on one number.

## Special tokens are real tokens

This is the section that seams directly into concept 04 (prompt roles), so it's worth being unambiguous about it now: when you interact with a chat model through a rendered UI, you see a clean back-and-forth of "you said / it said." What the model actually consumes is a single flat token sequence, and the role structure you see in the UI is represented inside that sequence by literal, ordinary vocabulary entries — control tokens like `<|user|>`, `<|assistant|>`, or a bracket-style pair like `[INST]` / `[/INST]` — that get inserted around each message's content before the sequence is tokenized [S2]. Hugging Face's chat template docs make the "these are just tokens, and the harness decides how" point unusually concretely: Mistral-7B-Instruct and Zephyr-7B are both fine-tuned from the *same* underlying Mistral-7B base model, but they use entirely different control-token schemes to mark the same role structure — one renders a user turn as `[INST] ... [/INST]`, the other as `<|user|>\n...`. Same base weights, different chat template, different literal tokens in the sequence [S2]. There is no universal "role" concept baked into the architecture; it's populated by whichever control tokens that specific model's chat template — and by extension, its tokenizer vocabulary — was fine-tuned to expect.

Two things follow from this. First, get the control tokens wrong — or reformat a chat with the wrong template — and you don't get a "close enough" response: performance degrades sharply, because the model is pattern-matching against a specific sequence of token IDs it saw at training time, not against some abstract sense of "this looks like a user turn" [S2]. Second, and this is the cost point: these tokens are real vocabulary entries. Every `<|user|>`, every BOS, every EOS, every generation-prompt marker appended to signal "now the assistant speaks" consumes a slot in the token sequence and therefore counts against the context window and the token-metered price, exactly like a token from the visible text does — even though none of it is visible in a rendered chat bubble. A harness that naively appends its own BOS/EOS on top of what a chat template already inserted doesn't just waste a token, it can actively hurt model performance by duplicating tokens the model wasn't trained to see doubled [S2]. The chat UI's job is precisely to hide this scaffolding from you; the token budget doesn't get that same courtesy.

## Why tokenization drives cost

Because pricing and context limits are counted in tokens rather than words or characters, the token-per-character rate of whatever you're sending directly determines what you pay and how much room you have left. English prose is the tokenizer's best case — its common words and morphemes were exactly what the training corpus was full of, so it compresses efficiently, roughly 0.75 tokens per word. Code, CJK scripts, and rare or out-of-distribution strings do not get that benefit: a byte-level vocabulary sized and merged primarily on English-heavy web text will represent less-common scripts and syntaxes with more tokens per character, because fewer of their common patterns made it into the learned merge table's high-frequency slots. The practical upshot is that "how long is this input" is not a character count or a word count question — it's a question about how well this particular text matches the statistics the tokenizer was trained on.

## Odd model behavior explained by tokenization

A handful of behaviors that look like model weirdness are really tokenizer artifacts, once you know to look for them.

Numbers split unpredictably. Because merges are frequency-driven rather than rule-driven, a number like 127 might land as a single token while 677 splits into " 6" and "77" — there's no principled reason for the difference beyond "that's where the frequency counts landed" [S1]. This is a big part of why LLMs are historically unreliable at arithmetic on multi-digit numbers: the model isn't manipulating digits, it's manipulating whatever arbitrary chunk the tokenizer handed it.

Leading whitespace changes the token. `" hello"` (with a leading space) and `"hello"` are different vocabulary entries with different token IDs — whitespace is part of the string and gets tokenized along with everything else, even though it's usually stripped from visualizations for readability [S1]. This is why prompt formatting details that look cosmetic — an extra space, a newline placed one way versus another — can measurably change model output: they're not cosmetic to the tokenizer, they're a different symbol.

The model can't reliably spell or count letters. Karpathy's framing of this is blunt and correct: "Why can't LLM spell words? Tokenization." [S1] The model never sees characters as its atomic unit of input — it sees whatever subword chunk the merge table produced. Asking it to reverse a word or count a letter inside it is asking it to reason about a decomposition it was never shown; the information is there in the byte sequence conceptually, but the model's actual input representation has already thrown away the character-level view.

## Pro vs. amateur

Amateurs think in words and characters; pros think in tokens, because that's the actual unit everything — cost, context budget, model behavior — is measured in. A prompt that "looks short" in words can still burn through a meaningful fraction of a context window if it's dense in rare strings, code, or non-English script.

Amateurs assume tokenization is a fixed, universal fact about a string. Pros know it isn't: the same text produces different token counts on different tokenizers, and a model upgrade can silently change how many tokens your existing prompts cost — recounting against the model you're actually going to use, rather than reusing a stale count from an earlier tokenizer, is a real and recurring practice, not paranoia. Different model families ship different named encodings for exactly this reason [S4], and the fact that this differs by model is itself the kind of detail an amateur budget calculation skips.

Amateurs think of a chat prompt as "the messages." Pros know a chat prompt is "the messages plus the chat template's control tokens," and that those control tokens are real, invisible-in-the-UI cost — every `<|user|>`, every generation-prompt token, every BOS/EOS the template inserts is billed and budgeted exactly like visible text [S2]. Getting the template wrong isn't a formatting nitpick either; it's the difference between the model correctly recognizing "it's my turn to answer" and the model continuing the user's sentence instead [S2].
