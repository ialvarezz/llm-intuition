# 05 · Context Window

## The window is the world

Everything a model can see about your conversation — the system prompt, every prior message, any documents you've pasted in, and the reply it's currently writing — has to fit inside one fixed budget of tokens. Anthropic's own framing is blunt about what that budget actually is: "the 'context window' refers to all the text a language model can reference when generating a response, including the response itself. This is different from the large corpus of data the language model was trained on, and instead represents a 'working memory' for the model" [S1]. That last phrase is the important one. The window isn't a cache of things the model has learned — it's the *entire* set of information available to it for this one response. Nothing outside it exists as far as the model is concerned.

It's also a hard cap, not a soft guideline. Feed a model more input tokens than its context window holds and the request fails outright — the API "returns a 400 `invalid_request_error` ('prompt is too long') on every model" [S1] once the input alone exceeds the limit. There's no graceful degradation at the input stage; the window is a wall, not a slope.

## Memory is re-sending

Chat apps feel like they remember you. They don't — the model has no memory between calls at all. What actually happens is that the app quietly re-sends the entire conversation so far, every single turn, and the model reads it fresh each time as if for the first time. This is why token counts keep climbing turn after turn: "as the conversation advances through turns, each user message and assistant response accumulates within the context window, and previous turns are preserved completely" [S1]. Turn 20 of a conversation costs more to process than turn 1, because turn 20's request silently includes turns 1 through 19 as well. The illusion of a model that recalls your name from three messages ago is really an app faithfully replaying the whole transcript back to a model that has no memory of ever seeing it.

## What happens at the limit

Two different things can happen once a conversation approaches the ceiling, and they're worth keeping separate. At the raw API level, if you try to send more input than the window allows, the request is simply rejected [S1]. But consumer chat interfaces work around this differently — rather than erroring out, they trim the conversation to keep it usable: "chat interfaces such as claude.ai can also manage the context window on a rolling 'first in, first out' basis" [S1]. In plain terms, the oldest turns quietly fall off the back of the window to make room for new ones. Nothing announces this in the UI. You just start noticing the model no longer reacts to something you told it early on — because that turn, literally, is no longer part of what got sent.

There's a subtler failure mode too, on generation itself: if a request's input plus the space reserved for the reply would blow through the window, some models will accept the request and then simply stop generating mid-answer once they hit the ceiling, returning a status that flags the response was cut off by the window rather than because the model chose to stop [S1]. Either way — eviction on the way in, or truncation on the way out — the lesson is the same: the window doesn't expand to accommodate you. Something gives.

## The KV cache

Here's a detail that explains a lot about how LLM pricing and latency actually behave. When a model generates a token, attention doesn't just look at the newest word — it recomputes relationships against *every* earlier token in the context, every single time. Doing that from scratch for the full prompt on every new token would be wasteful, since most of that earlier context hasn't changed between one generated token and the next. So implementations cache the intermediate representations (the keys and values from note 04's attention mechanism) for tokens already processed, and reuse them instead of recomputing.

This caching is exactly why providers can offer cheap re-use of a prompt prefix. Anthropic's prompt caching checks whether a matching prefix was already processed on a recent request; if not, "it processes the full prompt and caches the prefix once the response begins" [S3]. The saving is substantial and it comes from a very concrete place: "cached prefixes reuse the prior computation from when the prefix was first processed. The model doesn't need to re-process these tokens—it simply retrieves the cached intermediate representations. This elimination of redundant computation is why cache reads cost only 10% of the base input token price" [S3]. That 10%-of-base-price figure is a direct, quantified consequence of not having to redo the query/key/value math for tokens the model has already seen.

It also explains why a long prompt costs more the *first* time you send it than the second: the first pass has to build the whole cache from nothing (a full time-to-first-token cost across the entire input), while a second call reusing the same opening (say, a long system prompt or a big pasted document) can skip straight to reading from cache for that shared prefix and only pay full price for what's new. Caches aren't permanent, either — by default a cache entry has "a 5-minute lifetime" [S3] before it expires and the next request has to rebuild it.

## Lost in the middle

A bigger window is not the same thing as reliable recall across that window. A widely-cited study found that model accuracy on finding a specific fact depends heavily on *where* in the context that fact sits: "performance is often highest when relevant information occurs at the beginning or end of the input context, and significantly degrades when models must access relevant information in the middle of long contexts, even for explicitly long-context models" [S2]. The paper's own framing of the core result: "we find that performance can degrade significantly when changing the position of relevant information, indicating that current language models do not robustly make use of information in long input contexts" [S2]. In other words, a model given a 50-page document and asked a question whose answer sits on page 25 is measurably more likely to miss it than if the same fact sat on page 1 or page 50 — and this held true even for models built specifically to handle long contexts, not just older short-context ones.

Anthropic's own engineering guidance names a related, broader version of this problem: "as the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases," a phenomenon they call "context rot," which "affects all models" to varying degrees [S4]. The headline 128k-context or 1M-context number on a model card tells you the *ceiling*, not the *quality* of attention across that ceiling. Those are two different specs, and marketing tends to advertise only the first one.

## Pro vs. amateur

**Amateurs think of the context window as room for the prompt alone. Pros know the output has to fit inside the same budget.** The window "holds the conversation history plus the new output Claude generates" [S1] — every token the model writes back to you is drawn from the same pool as everything you sent it. Max out the input with a giant pasted document and there may be little or no room left for the model to actually answer. A maxed-out prompt isn't safely maxed-out; it's a prompt that's quietly starved its own reply of space.

**Amateurs put critical instructions once, at the top, and assume that's enough. Pros know instruction strength isn't uniform across a long context.** The lost-in-the-middle effect [S2] isn't only about retrieving facts from documents — the same positional weakness applies to instructions buried early in a long conversation or system prompt. If a constraint really matters, restating it near the end of a long context, close to where the model is about to generate, is a hedge against it getting diluted by everything that piled up in between.

**Amateurs treat pasting the whole document in as the free option. Pros know context is priced and quality-limited, so retrieval is often cheaper and better.** Every token of a pasted document counts against the same budget and the same per-token price as everything else in the request — there's no discount for tokens the model is merely skimming past. Anthropic's own agent guidance recommends the opposite of front-loading: rather than pre-processing all relevant data up front, "maintain lightweight identifiers (file paths, stored queries, web links, etc.) and use these references to dynamically load data into context at runtime using tools" [S4] — pull in only what's actually needed for the current step, instead of stuffing everything in up front just in case. That's the retrieval (RAG) pattern in miniature, covered properly in Module 2 — but the reason it exists at all traces straight back to this note: context is a finite, metered, and imperfect resource, not a bottomless notebook.

**Amateurs read a 128k-context spec as 128k tokens of equally good attention. Pros know the number on the box is a ceiling, not a quality guarantee.** A larger window changes what *can* fit; it says nothing about how evenly the model attends across all of it. Context rot and lost-in-the-middle [S2][S4] are properties of *all* models tested so far, long-context ones included — the honest way to read a 1M-context spec is that up to 1M tokens can be included, not that every one of those tokens gets read as carefully as the first.
